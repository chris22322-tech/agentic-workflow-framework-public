!!! note "Why 11 steps, not fewer"
    The most common mistake in scoping engineering workflows is collapsing data retrieval and classification into a single step. "Gather PRs and categorise them" hides the fact that fetching PRs (Step 2) is pure API retrieval with zero ambiguity, while categorising them (Step 4) involves label parsing, conventional commit prefix matching, and an LLM fallback -- three different strategies applied in a priority chain. Similarly, "generate release notes" could describe Steps 6-9 as a single step, but splitting them reveals that summary extraction (Step 6) and migration note generation (Step 9) are independent operations with different triggering conditions (Step 9 only runs if breaking changes exist). Each step has one action, one input, one output, and a clear boundary assignment. If you cannot assign a single boundary tag to a step, it is hiding two steps.

### Error Path Register

The workflow map above captures what each step does when it succeeds. The register below captures what happens when a step fails — specifically, which downstream steps are affected and how the workflow degrades. One row per failure condition, not per step.

| Step | Failure Category | Error Condition | Downstream Impact | Fallback Behaviour |
|---|---|---|---|---|
| 1 — Identify release scope | Tag resolution failure | The previous release tag cannot be resolved — `git describe --tags --abbrev=0 HEAD~1` returns no result and the GitHub Releases API has no prior release, or the current tag does not match the semver pattern | Abort — without a valid tag pair, Step 2 has no commit range to query. All downstream steps (2–11) cannot execute because the PR list that drives the entire workflow does not exist | Surface the error with diagnostic context (tag names attempted, API responses). Do not attempt partial processing — release notes with an undefined scope would be worse than no release notes |
| 2 — Fetch merged PRs | API rate limit | GitHub API returns HTTP 429 with `X-RateLimit-Reset` header during PR retrieval or commit-to-PR resolution | Step 2 cannot complete PR retrieval. All downstream steps (3–11) stall because the PR list is incomplete | Wait for the rate limit reset window and retry. If the reset window is >5 minutes, escalate to the engineer with rate limit context so they can decide whether to wait, provide a cached PR list, or use a higher-limit authentication token |
| 2 — Fetch merged PRs | API failure | GitHub Compare API returns HTTP 404 (tag not found in remote) or HTTP 500 on the `/repos/{owner}/{repo}/compare/{base}...{head}` endpoint | Abort — without a commit range, there are no merge commits to resolve to PRs. The same cascade as Step 1's tag resolution failure: Steps 3–11 have no input | Surface the error with diagnostic context (tag names, HTTP status, API response). If the failure is a 404, suggest verifying that both tags have been pushed to the remote repository |
| 3 — Fetch linked issues | Issue tracker API failure | Issue tracker API (GitHub Issues, Jira, or Linear) returns an HTTP error or times out on individual issue lookups | Affected PRs carry forward with empty or incomplete linked-issue metadata. Step 4 loses issue-type data that helps disambiguate LLM classification (strategy 3 uses linked issue types as a signal). Step 6 loses linked issue titles that inform LLM summarisation. Step 9 loses issue context that can enrich migration note generation for breaking change PRs | Store an error marker in the affected PR's `linked_issues` field; continue with available data. PRs with missing issue metadata proceed through Steps 4–9 with degraded context. Note the gap in the review output so the engineer can enrich manually during Step 10 |
| 4 — Categorise changes | No labels or commit prefix | A PR has no category-matching labels and its title or first commit message does not use a conventional commit prefix — neither strategy 1 (label match) nor strategy 2 (commit prefix) produces a classification | Step 4 falls back to LLM classification (strategy 3), which is probabilistic rather than deterministic. A miscategorised PR appears in the wrong section of the draft (Step 8). Step 10's review burden increases proportionally to the number of LLM-classified PRs | LLM classification with the PR title, description (first 500 characters), and linked issue types. Log the PR number to `llm_classified_prs` so the reviewer can scrutinise these entries during Step 10. If the team's labelling rate is low, this fallback handles the majority of PRs — the log serves as a feedback signal to improve labelling discipline |
| 6 — Extract change summaries | Empty or vague description | A PR has an empty or null `body` field, or a normalised title too vague to be informative (fewer than 4 words after stripping prefixes, or generic titles like "Fix bug" or "Update code") | Step 6 produces a lower-quality summary that the reviewer must rewrite in Step 10. Step 8 includes this summary in the draft release notes, where it may read as uninformative to end users | Apply the fallback chain: (1) check for a "Release Notes" section in the PR description — if present, use verbatim; (2) normalise the PR title by stripping prefixes and ticket references; (3) if the normalised title is too vague, send to LLM for summarisation from title, description, and linked issue titles. Log PRs that required LLM summarisation as candidates for PR template improvements |
| 9 — Generate migration notes | Sparse PR description | A breaking change PR has no "Migration," "Upgrade Guide," or "Breaking Change Notes" section in its description, and the description body is too sparse to infer specific migration steps | Step 9's LLM-generated migration instructions are generic rather than specific — "update your code to use the new API" instead of concrete before/after examples. Step 10's reviewer must write the migration instructions from scratch rather than editing the agent's draft | Send the PR title, description, and diff summary to the LLM with the migration instruction prompt. Accept the LLM's output even if generic — a generic migration note that the reviewer rewrites is more useful than no migration note at all. Flag sparse-description breaking change PRs in the review output so the engineer prioritises reviewing their migration notes |
| 2 — Fetch merged PRs | Empty result | The tag pair resolves to the same commit (e.g., a tag was recreated on an already-tagged commit) or the commit range contains no merge commits — `merged_prs` is empty after fetch | Steps 4–9 have no input to process. The entire categorisation, summarisation, and draft generation pipeline produces empty output. Step 10 receives a draft with no content to review | Generate a minimal release notes document stating "No changes in this release" and proceed to Step 10. The engineer decides whether to publish the empty release notes, add manual entries for changes the agent could not detect, or abort the release |

!!! note "Traceability: Error Path Register → Design error handling"
    The failure modes documented here are the source material for the error handling specifications in [Stage 4: Design](design.md). The register captures *what can go wrong and what it affects*; the design stage translates those into *how the agent handles it* — retry logic, error markers in memory, conditional routing, and escalation via HIL checkpoint. Register entries map to rows in the design's Error Handling Design table: Step 1's tag resolution failure and Step 2's compare-range failure consolidate into the design's "source-control API failure (tag/compare)" abort trigger — the design treats these as a single failure class because the downstream impact is the same (no commit range, no PRs to process). Step 2's rate limit maps directly to the design's "source-control API rate limit" row. Step 3's issue tracker failure maps to "Issue tracker API failure." Step 4's classification fallback maps to "PR with no labels and no conventional commit prefix." Step 6's empty description and Step 9's sparse description consolidate into the design's "PR with no description" row — the design abstracts these into a single pattern because the root cause (insufficient PR body content) is the same. Step 2's empty result maps to "Empty release (no PRs in range)." Three design-level concerns — ambiguous categorisation (model low confidence), malformed model output, and squash/rebase deduplication — do not appear in this register because they arise at the implementation level rather than the workflow level; they are addressed directly in the design stage. A fourth, breaking change missed by automation, is a process gap rather than a step failure — it is documented in Constraints and Assumptions, and the engineer's review in Step 10 is the safety net. The register ensures failure handling is designed from scope-level analysis, not improvised during build.

### Integration Requirements

- **Agent framework** — whichever orchestration framework your team has chosen (see [Getting Started: Choose a Platform](../getting-started/choose-a-platform.md))
- **Tool wrappers** — for the source-control platform, issue tracker, and CI/CD system your organisation uses
- **Language model provider** — for three tasks: ambiguous PR classification (Step 4), change summary generation (Step 6), and migration note generation (Step 9). These are the only steps that require model reasoning; the rest operate on structured data with deterministic logic
- **File output** — Markdown generation for the release notes document and migration guide
- **Credentials management** — environment variables or secrets manager for source-control token, issue tracker API key, and CI/CD API token

### Constraints and Assumptions

- **Entity cardinality:** The agent processes one release at a time. A release is defined by a pair of Git tags (previous and current). If the project uses a monorepo with multiple independently versioned packages, each package's release is a separate workflow run with its own tag pair. The agent does not cross-reference changes across packages -- if a PR touches two packages, it appears in each package's release notes independently. Monorepo coordination (e.g., ensuring consistent versioning across packages) is out of scope for v1.

- **PR labelling discipline is the primary data quality dependency.** Steps 4-5 rely heavily on PR labels for categorisation and filtering. If the team does not consistently apply labels, the agent falls back to conventional commit prefixes (Step 4, strategy 2) and then to LLM classification (Step 4, strategy 3). Each fallback layer is noisier than the previous one: labels are near-deterministic, commit prefixes require developer discipline, and LLM classification is probabilistic. The agent logs every PR that required fallback classification, creating a feedback loop: after each release, the team can review which PRs lacked labels and improve the process. If your team's labelling rate is below ~70%, expect the LLM classification step to handle the majority of PRs, which increases token cost and introduces categorisation variance between runs.

- **Squash-merge vs merge-commit vs rebase-merge:** The PR resolution strategy in Step 2 handles all three GitHub merge strategies, but the data quality differs. Squash-merged PRs map cleanly: one merge commit → one PR. Merge-commit PRs also map cleanly: the merge commit references the PR. Rebase-merged PRs are the problematic case: each rebased commit is a separate commit in the target branch, and the GitHub API must be queried per-commit to resolve back to the originating PR. If the team uses mixed merge strategies, expect Step 2 to make more API calls (one per rebased commit) and to require deduplication. A team that exclusively squash-merges will have the cleanest data path.

- **Breaking change detection has a gap.** The agent detects breaking changes via two signals: the `breaking-change` label (Step 4, strategy 1) and the `BREAKING:` / `BREAKING CHANGE:` prefix in commit messages (Step 4, strategy 4). Both require the PR author to explicitly flag the breaking change. If the author forgets both, the agent will not detect it. The LLM classification fallback may catch some obvious cases (e.g., a PR titled "Remove deprecated v1 endpoint"), but it cannot reliably detect breaking changes from titles alone. This is a known gap, not a solvable one at the agent level -- it is a process problem, and the engineer's review in Step 10 is the safety net. If breaking change misses are frequent, consider adding a heuristic: flag PRs that modify public API route definitions, change function signatures in exported modules, or alter configuration schema files.

- **Temporal scope:** The agent assumes that the release tag pair defines a clean, contiguous range of changes. If the team cherry-picks commits between releases (e.g., hotfixing a patch release from a feature branch), the commit range between tags may include commits that were already covered in a prior release's notes. The agent does not deduplicate against prior release notes -- if a commit appears in both `v2.3.1` (hotfix) and `v2.4.0` (next release), it will appear in both sets of release notes. The engineer should catch duplicates during the HIL step.

- **Processing model:** The workflow is sequential from Steps 1-3 (each step depends on the prior step's output). Steps 4-7 form a linear pipeline where each step transforms the output of the previous step. Steps 8 and 9 are independent of each other (both consume outputs from Steps 4-7 but do not depend on each other) and could execute in parallel. This parallelism is a design optimisation for [Stage 4](design.md), not a constraint -- the scope document captures the logical sequence.

- **Data sensitivity / governance:** The agent sends repository and development process data to an external LLM API for PR categorisation (Step 4), change summary generation (Step 6), and migration note generation (Step 9). The data categories sent to the LLM include: PR descriptions which may contain technical design rationale, security fix details, or references to vulnerabilities (Step 2), commit messages with author names and change context (Step 2), code review comments and discussion threads (Step 3, if enriched from issue trackers), issue and ticket content including bug reports with reproduction steps, feature specifications, and internal discussion (Step 3), and CI/CD build and test logs if used to enrich performance or security sections (Data Inventory). For most engineering teams, this data is classified as Internal. However, specific categories warrant attention: (1) security-related PRs may reference CVEs, vulnerability details, or exploit reproduction steps that should not be sent to external services before public disclosure; (2) PR descriptions for compliance or audit features may reference regulatory requirements or internal control details; (3) issue tracker content from Jira or Linear may include customer-reported bugs with PII or customer identifiers. **Governance prerequisites before build:** (1) Confirm the organisation's AI data processing policy permits sending source code metadata, PR descriptions, and issue tracker content to an external LLM provider. (2) Establish a convention for excluding security-sensitive PRs from LLM processing -- e.g., PRs with a `security` label bypass LLM classification and are manually categorised, or their descriptions are redacted before LLM input. (3) Complete the third-party risk assessment for the LLM provider. See the [Compliance Readiness pre-flight checklist](../stages/03-scope.md#compliance-readiness-start-early) in the methodology.

## Annotations

!!! note "Why Steps 1-3 are fully automated"
    These steps are pure data retrieval. Step 1 detects a trigger event (tag creation). Step 2 queries the GitHub API for merged PRs in a commit range. Step 3 resolves issue tracker references found in PR descriptions and commit messages. There is no judgement involved -- the agent is executing API queries with deterministic parameters. If the API call succeeds, you have the data. If it fails (rate limit, network error, missing permissions), you have an error to handle. This is the same pattern as the CSM's Steps 1-4 (pull CRM data, pull tickets, pull usage metrics, pull comms context) and the BA's Steps 1-5 (receive request, normalise, search backlog, pull roadmap context, pull architecture docs). Across all three worked examples, pure data retrieval is always AUTOMATE.

!!! note "Why Step 4 (categorisation) is AUTOMATE despite involving LLM classification"
    This is the most counterintuitive boundary decision in the workflow. The categorisation step uses an LLM for ambiguous PRs -- doesn't that require human judgement? The answer is the same as the BA's Step 3 (duplicate detection): categorisation here is a *flagging* step, not a *deciding* step. The agent proposes a category; the engineer validates it in Step 10. A miscategorised PR (listing a bug fix under "New Features") is caught in seconds during the review step. The cost of a wrong classification is a minor correction during review, not a wrong decision shipped to users.

    The three-strategy priority chain (labels → conventional commits → LLM) also limits the LLM's exposure. In a well-labelled repo, 70-80% of PRs are categorised by deterministic label matching alone. The LLM only handles the long tail of unlabelled, prefix-less PRs. If your team's labelling rate is high, this step is almost entirely rule-based. If it is low, the LLM does more work -- but the reviewer still catches errors. The boundary would only move to HIL if categorisation errors had consequences beyond "the reviewer fixes it" -- for example, if the release notes were auto-published without review.

!!! note "Why Steps 4-9 are all AUTOMATE -- the metadata advantage"
    The [Select stage](select.md) introduced the metadata/meaning distinction as the strongest predictor of automation success for engineering workflows. This is where that prediction materialises in the boundary assignments. Every step from 4 to 9 operates on *metadata* -- PR labels, commit prefixes, issue types, file change counts, template structures -- not on the *meaning* of the code changes themselves. The agent never needs to understand what a code change does; it only needs to read the structured data *about* the change.

    Compare this to the CSM example, where Steps 5-8 involve interpreting sentiment, assessing context, and synthesising qualitative signals -- operations on *meaning* that require HIL checkpoints. Or the BA example, where Step 7 (strategic alignment) requires understanding organisational context that resists documentation. In the SE example, the hardest analytical step (categorisation) is still operating on labels and commit messages, not on diffs or code semantics. This is what makes Release Notes Compilation score Complexity 5 in the selection stage, and it is what allows a continuous run of AUTOMATE tags from Step 1 through Step 9.

    The absence of HIL checkpoints before Step 10 does not mean these steps are trivial. It means their error modes are all catchable in a single review step, rather than requiring checkpoints along the way. This is the "metadata advantage" in practice: when the agent operates on structured data, its errors are *classifiable* (wrong category, missed label, vague summary) rather than *interpretive* (wrong sentiment reading, misjudged political context), and classifiable errors are cheaper to catch in batch review.

!!! note "Why Step 10 (review and refine) is the single HIL checkpoint"
    The CSM example has four HIL checkpoints scattered across the workflow (Steps 5, 6-7 edge cases, 8, 9-11). The BA example has three (Steps 7, 9-10). This example has one. That is not because the SE workflow is simpler -- it is because the error characteristics are different.

    In the CSM workflow, a wrong sentiment reading in Step 5 propagates through every downstream step: it skews the risk assessment, distorts the health score, and contaminates the report. Catching it early (at Step 5) is far cheaper than catching it late (in the final report). In the BA workflow, a wrong strategic alignment score in Step 7 feeds directly into the priority recommendation in Step 9 -- again, an early checkpoint prevents compounding errors.

    In the release notes workflow, categorisation errors in Step 4 do not compound. A miscategorised PR appears in the wrong section of the draft, but it does not cause other PRs to be miscategorised or summaries to be rewritten. Each PR's classification, summary, and position in the document are independent. This means all errors can be caught in a single pass during Step 10, rather than requiring intermediate checkpoints. The engineer reads through the draft, spots the miscategorised PR, moves it to the correct section, fixes any vague summaries, and approves. One focused review is more efficient than three scattered checkpoints for this workflow's error profile.

!!! note "Why Step 11 is MANUAL -- but could become AUTOMATE"
    Step 11 is tagged MANUAL because publishing is the point of no return: once the release notes are posted to the GitHub Release page, they are visible to users. The engineer needs to decide *when* to publish (immediately, or after the deploy is confirmed successful), *where* to publish (GitHub Release only, or also Slack, email, docs site), and *how* to frame the announcement (routine release vs. security patch requiring urgent attention).

    However, this is the step most likely to shift boundaries over time. If the team adopts a convention where every release is published immediately after approval, and the distribution channels are always the same (GitHub Release + CHANGELOG.md commit), then publishing becomes a deterministic API call with no judgement involved. The boundary would shift from MANUAL to AUTOMATE, and the entire workflow would have a single HIL checkpoint at Step 10 with everything else automated. That is a valid end state -- but start MANUAL until the team confirms the publishing decision is always the same.

    This mirrors the CSM example's Step 12 (distribution) and the BA example's Step 11 (submission to prioritisation committee). In all three worked examples, the final distribution step starts MANUAL because the consequences of a premature or misdirected publication are difficult to reverse. The CSM sends a report to stakeholders; the BA submits to a committee; the SE publishes to users. All three require human judgement about timing, audience, and framing -- at least until the pattern stabilises.

!!! note "Traceability: Decision Logic → Stage 4 action specs → Stage 5 config"
    The Decision Logic entries for Steps 4, 5, 7, and 9 are the source material for action specifications in [Stage 4](design.md). The label-to-category mapping (Step 4), the exclusion label list (Step 5), the section ordering and impact sort heuristics (Step 7), and the breaking change migration template (Step 9) captured here translate directly into externalised config that the agent loads at runtime. If you trace the criteria forward to the Design stage, you should find them unchanged -- capture once in Stage 3, externalise as config in Stage 5. The categorisation priority chain (labels → conventional commits → LLM) becomes the action's processing logic; the exclusion labels become a config list; the section ordering becomes a config constant; the migration note format becomes a prompt template.

!!! tip "The boundary should err toward more human involvement in early versions"
    Step 4 (categorisation) is the most likely candidate to gain an HIL checkpoint if the current boundary proves too aggressive. If the team's PR labelling discipline is poor and the LLM classification fallback produces frequent errors, the review burden in Step 10 may become heavy enough that a checkpoint after Step 4 -- "review the categorisation before the agent generates summaries and drafts notes from it" -- would be more efficient than catching all errors at the end. Similarly, Step 9 (migration notes) could warrant its own HIL checkpoint if breaking changes are frequent and the migration instructions are safety-critical (e.g., database migration steps that could cause data loss if documented incorrectly). The current single-checkpoint design assumes that the team's labelling is reasonable and that breaking changes are infrequent enough to review in context. If either assumption breaks, add a checkpoint rather than tolerating review fatigue.
