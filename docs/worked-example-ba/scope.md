### Error Path Register

The workflow map above captures what each step does when it succeeds. The register below captures what happens when a step fails — specifically, which downstream steps are affected and how the workflow degrades. One row per failure condition, not per step.

| Step | Failure Category | Error Condition | Downstream Impact | Fallback Behaviour |
|---|---|---|---|---|
| 1 — Receive feature request | Trigger missed | Jira webhook fails to fire, Slack message does not match the expected request format, or email is filtered before reaching the shared inbox | Silent failure — Steps 2–11 never execute because the workflow is never triggered. The feature request sits unprocessed until someone notices manually | No automated fallback within the workflow. Configure alerting on trigger channel health (webhook delivery logs, Slack channel activity, inbox filtering rules). If the trigger mechanism is unreliable, add a manual fallback: periodic scan of request channels by the BA to catch missed requests |
| 2 — Normalise and log in Jira | System failure | Jira API returns error or times out during ticket creation | Abort — all downstream steps depend on the Jira ticket. Without a normalised ticket, there is no `request_details.component` to drive duplicate detection (Step 3), no component to look up in the service catalogue (Step 5), and no historical data to query (Step 6). An error marker here would cascade through every subsequent step, producing an impact assessment about nothing | Surface the error with diagnostic context (HTTP status, Jira instance URL, timestamp). Do not attempt partial processing. Alert the BA to create the ticket manually and re-trigger the workflow |
| 3 — Search backlog for duplicates | Data unavailable | Jira search API returns error or times out when querying the backlog | Step 9 draft assessment has no duplicate analysis section — the prioritisation committee receives an assessment without knowing whether this work has already been requested. ### Integration Requirements

- **Agent framework** — whichever orchestration framework your team has chosen (see [Getting Started: Choose a Platform](../getting-started/choose-a-platform.md))
- **Tool wrappers** — for the ticketing platform, wiki, roadmap tool, and OKR tracker your organisation uses
- **Language model provider** — for semantic duplicate matching, strategic alignment assessment, impact summary generation, and priority recommendation
- **Embedding model** — for semantic similarity in duplicate detection (Step 3); can use the same model provider or a dedicated embedding endpoint
- **File output** — Markdown or structured JSON for the impact assessment document
- **Credentials management** — environment variables or secrets manager for API tokens (ticketing, wiki, roadmap, OKR tracker)
- **Webhook listener** (optional) — for real-time ticket creation trigger instead of polling

### Constraints and Assumptions

- **Entity cardinality:** The agent processes one feature request at a time. If multiple requests arrive simultaneously (e.g., three requests submitted in the same Slack thread), each is processed independently as a separate workflow run. The agent does not batch or compare concurrent requests against each other — if two competing requests arrive the same day, each assessment is written without knowledge of the other. The prioritisation committee resolves conflicts across assessments.

- **Temporal / frequency:** The workflow assumes OKRs and roadmap items are current as of the time of execution. OKRs are set quarterly; the agent treats all OKRs with "active" status as current. If OKRs are not updated when a new quarter begins (stale data), the alignment assessment in Step 7 will evaluate against outdated objectives. The agent does not validate OKR freshness — if the tracker shows Q4 OKRs as active in Q1, it will use them. This is a data quality dependency, not a technical one.

- **Access / permissions:** The agent requires read access to all Jira projects whose backlogs it searches for duplicates. In organisations with restricted project visibility, the agent's service account may miss duplicates in projects it cannot access. The BA should confirm that the service account has `browse_project` permission across all relevant Jira projects before build. Confluence access assumes the agent can read the spaces containing architecture docs, ADRs, and planning decision records.

- **Stability / change rate:** The complexity estimation model (Step 6) depends on historical story point data being consistent — if the team recalibrates its story point scale mid-quarter, historical comparisons become unreliable. The strategic alignment criteria depend on OKRs being specific and measurable; vague OKRs ("improve customer satisfaction") produce vague alignment scores. The impact assessment template is assumed stable; if the prioritisation committee changes its expected format, the template in Step 9 must be updated in the agent's config.

- **Processing model:** The workflow is sequential — each step depends on the outputs of prior steps. However, Steps 4 and 5 (roadmap context and architecture docs) are independent of each other and could execute in parallel. Steps 6, 7, and 8 all depend on earlier retrieval steps but are independent of each other and could also execute in parallel. This parallelism is a design optimisation for [Stage 4](../stages/04-design.md), not a constraint — the scope document captures the logical sequence, and the design stage determines execution order.

- **Data sensitivity / governance:** The agent sends internal product and organisational data to an external LLM API for duplicate detection, strategic alignment assessment, impact summary generation, and priority recommendation. The data categories sent to the LLM include: feature request details with requester names and business justification (Steps 1–2), user feedback and customer-reported problem descriptions (Step 1), backlog and prioritisation data including story point history and sprint planning context (Steps 3, 6), architecture documentation and service catalogue entries including dependency maps and ADRs (Step 5), roadmap and OKR content including strategic objectives and key results (Steps 4, 7), and stakeholder communications from Confluence planning pages that may reference commercial commitments, partner agreements, or internal strategic decisions (Step 4). While less sensitive than the CSM's client financial data, this content includes internal strategic priorities, product roadmap details, and organisational decision-making context that most organisations classify as Internal or Confidential. **Governance prerequisites before build:** (1) Confirm the organisation's AI data processing policy permits sending internal product strategy and backlog data to an external LLM provider. (2) Verify that architecture documentation and ADRs do not contain security-sensitive infrastructure details (e.g., internal IP ranges, authentication flows, vulnerability assessments) that should be excluded or redacted before LLM processing. (3) Complete the third-party risk assessment for the LLM provider. See the [Compliance Readiness pre-flight checklist](../stages/03-scope.md#compliance-readiness-start-early) in the methodology.

## Annotations

!!! note "Why Steps 1-5 are fully automated"
    These steps are data retrieval and transformation. Step 1 detects a trigger event. Step 2 normalises the input into a standard format. Steps 3-5 pull data from systems using structured queries. There is no judgement involved — the agent is executing queries with known parameters against systems with documented APIs. The only failure mode is technical (API down, malformed response, empty results), not interpretive. If the query succeeds, the output is correct. If it fails, the error path is clear: flag the gap, continue with available data.

    Compare this to the CSM example, where Steps 1-4 are also fully automated for the same reason — pure data retrieval. The pattern holds across roles: if a step is "open a system, run a query, return the results," it is AUTOMATE. The BA's data sources are different (Jira and Confluence instead of Salesforce and Zendesk), but the boundary logic is identical.

!!! note "Why Step 3 (duplicate detection) is AUTOMATE despite requiring LLM judgement"
    This might seem like it should be HIL — after all, the agent is making a judgement call about whether two requests are "the same thing." But duplicate detection in this workflow is a *flagging* step, not a *deciding* step. The agent surfaces candidates ranked by similarity score; it does not close tickets or merge requests. A false positive (flagging a non-duplicate) costs the BA ten seconds of dismissal during review in Step 10. A false negative (missing a true duplicate) means the committee sees two assessments for the same thing — annoying but not harmful, and the committee catches it.

    The risk calculus matters: when the cost of an incorrect agent judgement is "the human spends a few extra seconds reviewing," the step can be automated. When the cost is "a wrong decision ships to stakeholders," it needs a human checkpoint. Duplicate detection is the former.

!!! note "Why Step 6 (complexity estimation) is AUTOMATE and not a split boundary"
    The complexity estimation is criteria-based: count components, look up historical medians, check for schema changes and external dependencies, apply the tier mapping. Given the same inputs, any two BAs would produce the same tier (or within one tier of each other). This is not the same as *effort estimation* — which involves team dynamics, technical debt knowledge, and calendar awareness that the agent cannot access. The scope deliberately limits this step to *complexity tier* (a structural property of the request) rather than *effort estimate* (a prediction about how long it will take a specific team). That distinction is what keeps it fully automatable.

    The BA validates the complexity estimate during review (Step 10) and can override it. If the agent consistently over- or under-estimates for a particular component, that is a prompt refinement issue for iteration in [Stage 6](../stages/06-evaluate.md), not a reason to add a HIL checkpoint to every run.

!!! note "Why Step 7 (strategic alignment) splits between AUTOMATE and HIL"
    The scoring component is mechanical: for each OKR, compare the request description against the objective and key results, classify the relationship (Direct / Indirect / Neutral / Conflicting), and compute the alignment score. An LLM can do this reliably when the OKRs are specific and measurable — "Increase API response time p95 to under 200ms" is clear enough that the agent can assess whether a feature request contributes to it.

    The interpretation component is where the BA adds value. Strategic alignment in practice is not purely a function of what the OKR tracker says. A feature request might score "Low alignment" against documented OKRs but be informally tied to a sales commitment the CEO made last week. Another might score "High alignment" but conflict with an initiative that is not yet in the tracker because it was decided in yesterday's leadership meeting. The BA knows the organisational context that the OKR tracker does not capture.

    This is the same pattern as the CSM's split-boundary steps (ticket health, engagement analysis) — but applied to a different type of judgement. The CSM's splits separate *quantitative calculation* from *contextual interpretation of metrics*. The BA's split separates *documented strategy matching* from *organisational context that resists documentation*. Different domain, same principle: automate the part that operates on structured data, checkpoint the part that requires knowledge the agent cannot access.

!!! note "Why Step 10 is HIL and not MANUAL"
    The BA is not starting from scratch — the agent has produced a complete draft assessment with populated sections, a recommended priority, and flagged issues. The BA's job is to validate, correct, and enrich. This is the textbook HUMAN-IN-THE-LOOP pattern: the agent does 80% of the work, the human provides the 20% that requires domain knowledge and organisational context.

    In practice, the BA's review focuses on a few high-value additions: (1) catching false-positive duplicates that the semantic search surfaced, (2) adjusting complexity estimates based on knowledge of technical debt or team capacity that is not in Jira, (3) adding political context to the alignment assessment, and (4) sanity-checking the priority recommendation. On a clean request with good data, the review takes 5-10 minutes. On a complex cross-team request, it might take 20-30 minutes — but the agent has already saved the 60-90 minutes of data gathering and initial analysis.

!!! note "Why Step 11 is fully MANUAL"
    Submission is the point of no return. Once the assessment reaches the prioritisation committee, it represents the BA's professional judgement. Three decisions at this step require human context the agent cannot have: (1) *Where* to submit — the weekly triage meeting, an ad-hoc review, or a direct escalation. This depends on the request's urgency, the current committee calendar, and whether the BA judges it needs special attention. (2) *How* to frame it — some assessments benefit from a cover note that contextualises the request ("this is the third request in this area this quarter" or "the VP of Engineering mentioned this in last week's all-hands"). (3) *Who* to loop in — pre-briefing a stakeholder before the committee meeting is sometimes necessary to avoid surprises, and the BA is the person who knows when that is the case.

    This mirrors the CSM example's Step 12 (final review and distribution): the irreversibility of the action plus the dependence on relationship and political context makes it MANUAL. This is not a limitation — it is a deliberate safety boundary.

!!! note "Traceability: Decision Logic → Stage 4 action specs → Stage 5 config"
    The Decision Logic entries for Steps 3, 6, 7, and 9 are the source material for action specifications in [Stage 4](design.md). The duplicate similarity thresholds (0.65 and 0.80), the complexity tier mapping (component count × schema change × external dependency), the OKR alignment classification (Direct / Indirect / Neutral / Conflicting), and the priority recommendation formula (alignment × urgency) captured here translate directly into externalised config that the agent loads at runtime. If you trace the criteria forward to the Design stage, you should find them unchanged — capture once in Stage 3, externalise as config in Stage 5.

!!! tip "The boundary should err toward more human involvement in early versions"
    Step 7 (strategic alignment) is the most likely candidate to shift boundaries over time. If after several months of use, the BA finds they rarely override the agent's alignment assessment — because the OKRs are consistently well-maintained and the organisational context rarely changes the score — that HIL checkpoint can become fully automated. Conversely, if duplicate detection (Step 3) starts producing too many false positives that waste review time, it might warrant adding a lightweight HIL filter between Steps 3 and 4. The boundary is a starting position, not a permanent one. But you cannot easily *add* a checkpoint after users start relying on a fully automated step. Start conservative.
