Section 1 3 min read

What a personalised site looks like

The generic framework site teaches the methodology. A personalised site teaches the methodology using your company's language and context. The methodology stays the same—six stages, the same concepts, the same rigour. But every example, every template, every reference point is yours.

What changes

📄
Worked examples

Use your actual workflows instead of a generic CSM's health review. Your team reads examples they recognise.

🔧
Templates

Pre-fill with your systems—Jira, Salesforce, whatever tools your org uses. No more replacing placeholder names.

📖
Glossary

Includes your domain terms. If your org calls them "engagements" instead of "accounts," the site speaks your language.

🔌
Decision trees

Add your organisation's constraints as early gates. Compliance, data access, budget thresholds—built right in.

🚀
Quick Start

Uses your role as the example, so new team members can follow along with a scenario they will actually encounter.

What stays the same

The six-stage methodology. The core concepts. The underlying structure that makes the framework work. Personalisation wraps your context around the methodology—it does not change the methodology itself.

Section 2 3 min read

How it works

There are two paths to a personalised site. The GitHub Action path is simpler and handles the build automatically. The local path gives you more control. Choose whichever fits your team's workflow.

Recommended

GitHub Action

  1. Fork the framework repo. This gives you your own copy that you control, while still being able to pull updates from upstream.
  2. Fill in your org-config.yaml. This is the config you built in Layer 1—your company name, roles, systems, workflows, and domain terms.
  3. Push to GitHub. Commit your config file and push. That is the only manual step.
  4. The Action builds your personalised site automatically. GitHub Actions picks up the push, reads your config, generates personalised content alongside the generic pages, and deploys the site.
  5. Your team opens it and sees their world. Their company name, their roles, their workflows—all woven through the framework they are learning.
Advanced

Local build

  1. Clone the repo to your local machine.
  2. Run the fork script: python personalise/fork.py --config personalise/org-config.yaml
  3. The script generates personalised content alongside the generic pages. Your config drives every substitution and example generation.
  4. Preview with mkdocs serve to see the personalised site locally before sharing it with your team.
  5. Deploy wherever you host docs. GitHub Pages, Netlify, Vercel, an internal wiki—the output is standard HTML.
Both paths, same result

Whether you use the GitHub Action or the local build, the output is the same: a complete framework site where every generic reference has been replaced with your organisation's specifics. The GitHub Action path just automates the build-and-deploy step.

Section 3 2 min read

Keeping it updated

The upstream framework evolves. New stages get refined, new templates are added, examples improve. Your personalised site should stay in sync without losing your customisations.

📥
Pull upstream changes

Run git pull upstream main to bring in the latest methodology updates, new templates, and improved examples from the framework's main repository.

🔄
Re-run the fork

Your personalised content regenerates with the latest methodology. The config you already have drives the personalisation—no need to reconfigure anything.

🔍
Version drift detection

A built-in script tells you what changed upstream since your last sync. You can review the diff before regenerating, so there are no surprises.

If you are using the GitHub Action path, updates are even simpler. Pull upstream, push, and the Action rebuilds everything automatically. Your personalised site stays current with zero manual regeneration.
