Drawing the Line -- What the Agent Does vs What You Do

Every step in your workflow gets one of three tags. Get this right and you have a clear automation boundary. Get it wrong and you either waste time reviewing things that do not need review, or let the agent loose where it should not be.

5 / 8
1

The Three Tags

5 min

Every step in your workflow gets exactly one of three tags. No grey areas, no "it depends." One tag per step. Here they are:

Automate
The agent does it. You do not review.
Data retrieval, calculations, template-based generation. Deterministic work where the output is either correct or it is not.
  • Pull account data from Salesforce
  • Calculate ticket volume trends
  • Generate usage summary from analytics API
  • Format data into report template
Human-in-the-Loop
The agent does 80%. You review and approve.
The agent drafts, flags, or proposes. You check it before the workflow continues. This is your quality gate.
  • Agent drafts the health narrative -- you edit tone
  • Agent flags at-risk accounts -- you confirm priority
  • Agent proposes action items -- you approve or revise
  • Agent compiles renewal brief -- you add context
Manual
You do it entirely. The agent does not touch it.
Relationship-dependent work, political sensitivity, or anything that requires real-time human interaction.
  • Send the report to the customer
  • Lead the QBR conversation
  • Decide whether to escalate internally
  • Navigate a difficult renewal negotiation

The Golden Rule

3 min
Golden Rule

Err toward more human involvement in v1. You can always remove checkpoints later as trust builds.

It is tempting to automate aggressively from the start. Resist that temptation. Your first version should have more checkpoints than you think you need. Here is why, and here is how you know when to adjust:

A 95% approval rate at a checkpoint means the checkpoint is unnecessary.

If you are approving the agent's output without changes 19 times out of 20, that checkpoint is costing you time without adding value. Remove it in v2.

95%+ approval = remove checkpoint

A 60% edit rate means the preceding step needs improvement.

If you are editing the agent's output more often than not, the problem is not the checkpoint -- it is the step before it. The agent's instructions, data sources, or criteria need refinement. Fix the agent, not the checkpoint.

60%+ edits = fix the agent

This creates a natural evolution. Your v1 has many checkpoints. Over time, data shows which checkpoints are adding value and which are just slowing you down. You tighten the screws gradually, from evidence, not from hope.

Common Boundary Mistakes

5 min

These are the four mistakes we see most often when people draw the automation boundary for the first time. All four are completely natural. All four are fixable.

Mistake 1
Automating sentiment analysis completely

You tag "assess customer sentiment" as AUTOMATE because the agent can read NPS scores and survey responses. But sentiment is not just data -- it is relationship context. The NPS might say 8, but you know from last week's call that the champion is frustrated about a feature gap.

Keep sentiment assessment as HIL. The agent compiles the data signals. You interpret what they mean for this specific customer.
Mistake 2
Keeping data gathering as MANUAL

You tag "pull account data from Salesforce" as MANUAL because "I always do it myself" or "I want to make sure it is right." But data retrieval is the single most automatable category of work. If the query is well-defined, the agent will pull the exact same data you would, every time, in seconds instead of minutes.

Data gathering is always AUTOMATE. If you are worried about data quality, add a validation step after, not a manual step before.
Mistake 3
Making everything Human-in-the-Loop

You tag every step as HIL because it feels safe. But if you review every single thing the agent does, you spend more time reviewing than you would spend doing the work yourself. The overhead of context-switching into "review mode" for trivial outputs is real.

Reserve HIL for steps where your review actually changes the output. If you would rubber-stamp it, it should be AUTOMATE.
Mistake 4
Forgetting that distribution stays MANUAL

You build a beautiful automated workflow, and at the end you tag "send the report" as AUTOMATE. The agent sends an email to your customer with a tone you did not review, at a time you did not choose, with no opportunity to add a personal note. This is how trust erodes.

Distribution is always MANUAL. The agent prepares. You deliver. No exceptions in v1.
Pattern to notice

Mistakes 1 and 4 are about being too aggressive (too much automation). Mistakes 2 and 3 are about being too conservative (too much human involvement). The sweet spot is in the middle, and the decision tree from Section 2 is designed to land you there.

What You Now Have

You have a workflow with steps, decision logic classifications, and automation boundary tags. That is the core of a Stage 3 scope document. You know what the agent does, where you review, and what stays entirely with you.

This is your blueprint. If your team uses a low-code agent builder, this is what you will configure yourself. If you are working with an engineer using a code-first framework, this is the handoff document that tells them exactly what to build. Either way: no ambiguity, no assumptions, no "we'll figure it out later."

In Lesson 6, you will learn how to refine this into a complete scope document and prepare for the next stage — whether that is configuring the agent yourself or handing off to a technical partner.
