Scope the Workflow

You have selected your first workflow. Now you turn it into a precise specification that an engineer can build from — or that you can hand to an AI coding agent. Every gap in your scope is a gap in the agent's behaviour.

4 / 8
1

Why Scope Matters More for BAs

~5 min
Key Principle

For an engineer, the scope document is a planning tool. For a BA, the scope document IS the deliverable.

When a software engineer works through the framework, the scope document helps them plan what they are about to build. They will iterate on it as they code. But when you, as a BA, produce a scope document, you are handing it to someone else to build. That means your scope must be complete enough to stand on its own.

Think of it this way: every gap in your scope is a gap in the agent's behaviour. If you leave a decision undefined, the engineer has to guess — or come back and ask. If you leave an error path unspecified, the agent will do nothing when it encounters that failure.

A complete scope document has five components:

  1. Step-by-step workflow map — every action the agent takes, in order
  2. Decision logic — the explicit criteria for every non-trivial decision
  3. Automation boundaries — which steps are automated, human-reviewed, or fully manual
  4. Error paths — what happens when a step fails or data is missing
  5. Data inventory — every system the agent needs to access, and how

The rest of this lesson walks through each component using the feature request intake workflow from our worked example.

2

Workflow Mapping — The BA Advantage

~8 min

You already map processes. That is a core BA skill. The framework adds one thing on top of what you already do: explicit decision logic capture.

Most process maps describe what happens. An agentic workflow scope must describe how decisions are made — in enough detail that a machine can execute them. The difference between a good and bad scope is almost always in the decision logic.

Bad example: vague decision step

"Step 3: Assess the request's complexity."

This is how most process maps are written. A human reading this knows roughly what it means because they bring context. An agent reading this has nothing to work with. What factors determine complexity? What are the thresholds? What are the output categories?

This step is unimplementable. An engineer receiving this scope will either guess, make something up, or come back to ask you — burning time either way.

Good example: explicit decision logic

"Step 3: Map to complexity tier using four factors:"

(1) Components affected — 1 = Small, 2–3 = Medium, 4+ = Large.

(2) Historical story point baseline — pull median from last 4 sprints. Below median = Small, 1–2x median = Medium, above 2x = Large.

(3) Schema change required — if yes, add one tier.

(4) External integration — if yes, add one tier.

This is implementable. Every factor has a data source and a threshold. An engineer can build this without guessing. An LLM-based agent can apply these rules to structured data.

Key Test

If you cannot write the decision logic down, the step probably needs a Human-in-the-Loop boundary. That is not a failure — it is valuable information about where the human remains essential.

3

Drawing the Automation Boundary

~7 min

Every step in your workflow needs a boundary tag. There are three types:

AUTOMATE

The step applies documented criteria to structured data. The agent can execute it without human involvement.

Examples: duplicate detection via keyword and semantic matching, complexity tier calculation using the four-factor model, data retrieval from APIs.

HIL

The agent does the work, but a human reviews and approves before the workflow proceeds. Human-in-the-Loop.

Examples: strategic alignment interpretation (agent scores, human validates), impact assessment review, any output that will be shared externally.

MANUAL

The step depends on context that only a human has. The agent cannot meaningfully contribute.

Examples: submission to a governance committee, stakeholder pre-briefing, negotiation of competing priorities.
The Boundary Test

If the answer depends on criteria you can write down, it is AUTOMATE. If it depends on context you carry in your head, it is HIL or MANUAL.

Here is how boundary tags apply to the feature request intake workflow:

1
Receive feature request
Detect that a new request has been submitted via the intake channel (Jira ticket creation, Slack message, or form submission).
AUTOMATE
Trigger detection on structured events. No judgement required.
2
Search backlog for duplicates
Query Jira for existing tickets using keyword matching and semantic similarity. Flag potential duplicates above a configurable similarity threshold.
AUTOMATE
Keyword + semantic matching with documented thresholds. Data is structured, criteria are explicit.
3
Estimate complexity
Apply the four-factor complexity model: components affected, historical story points, schema change flag, external integration flag.
AUTOMATE
All four factors have data sources and thresholds. Criteria-based calculation.
4
Assess strategic alignment
Score the request against current OKRs and roadmap priorities. The agent produces a numeric alignment score; the human interprets whether that score reflects actual strategic intent.
AUTOMATE scoring / HIL interpretation
Scoring uses documented criteria (OKR keyword matching). Interpretation requires organisational context the agent does not have.
5
Review and refine assessment
BA reviews the complete draft assessment, corrects any errors, adds context from stakeholder conversations, and approves for distribution.
HIL
Validates context the agent cannot see: recent conversations, political dynamics, unstated priorities.

Notice that Step 4 has a split boundary. The scoring sub-step is automatable because OKRs are documented. The interpretation sub-step is HIL because understanding what leadership actually cares about (versus what the OKR tracker says) requires human context. Splitting a step like this is normal and expected.

4

Error Paths and Data Inventory

~5 min

Most scope documents stop at the happy path. But agents encounter failures all the time — APIs go down, data is stale, inputs are malformed. If you do not specify what should happen, the agent will either crash or silently produce incomplete output.

Error paths answer one question: when this step fails, what should the agent do?

Error example: Jira API fails during duplicate search

Failure: The Jira REST API returns a 5xx error or times out during the duplicate search step.

Specified behaviour: Store an error marker in the assessment output. Continue the workflow without duplicate analysis. Flag the gap clearly in the final output so the BA knows to check manually during the HIL review step.

Why this matters: Without this specification, the engineer must decide: should the agent retry? Abort? Skip silently? Each choice has different consequences for the BA's review workload. This is a business decision, not a technical one.

Error example: OKR tracker returns stale data

Failure: The OKR tracker returns data from the prior quarter because it has not been updated.

Specified behaviour: The agent cannot detect staleness — it has no way to know whether Q3 OKRs are still current or should have been replaced by Q4 OKRs. The BA validates OKR currency during the HIL review step. The scope document notes this as a known limitation.

Why this matters: Not every error can be handled automatically. Documenting that staleness detection is a human responsibility prevents the engineer from building a brittle freshness check that creates false confidence.

Data inventory lists every system the agent needs to access. This is the information an engineer needs to set up integrations.

Source Access Method Auth Notes
Jira REST API v3 OAuth 2.0 / API token Read access to project backlog; search endpoint for duplicates
Confluence REST API v2 OAuth 2.0 / API token Read access to requirements pages and decision logs
Roadmap tool REST API API key Read access to current roadmap priorities and timeline
OKR tracker REST API API key Read access to current quarter OKRs; staleness not detectable via API
BA Tip

You do not need to know the technical details of every API. Your job is to identify what the agent needs to access and why. The engineer will handle the how. But if you know the access method (e.g., "they have a REST API" vs. "we currently export CSVs manually"), include it — it saves a round of questions.

6

Self-Check

~3 min

Test your understanding with these reflection questions. Try to answer before expanding.

You have a step that says "determine if the request is urgent." What boundary tag should this get?

It depends. If urgency follows documented criteria — SLA breach detected, P1 keyword in the title, executive flag on the ticket — then the step is AUTOMATE. The agent can check those conditions against structured data.

If urgency depends on reading organisational politics — "the VP mentioned this in a hallway conversation" or "this team is already unhappy with us" — then it is HIL. The agent cannot access that context.

The key question is always: can you write the criteria down?

Why do we document error paths at the scope stage instead of waiting for build?

Error handling decisions are business decisions, not technical ones. When the Jira API fails, should the agent degrade gracefully and continue without duplicate data? Or should it abort and notify the BA? That choice affects the BA's review workload and the quality of the output.

The BA decides how the workflow should behave when data is missing — that is a requirement, not an implementation detail. If you leave it to the engineer, they will make a technical choice (retry 3 times, then throw an error) that may not match what the business actually needs (skip and flag the gap).

A step in your workflow scores as HIL. Does that mean the agent does nothing for that step?

No. HIL means the agent does the work and the human reviews it. The agent still contributes — it might draft the output, gather the data, or apply preliminary scoring. The human validates, corrects, and approves.

Only MANUAL means the agent has no meaningful role. HIL is a collaboration model, not an exclusion.

What you should have after this lesson

  • Complete step-by-step workflow map for your selected workflow
  • Decision logic documented for every non-trivial step
  • AUTOMATE / HIL / MANUAL boundary tags with justification for each step
  • Error paths specified for critical failure points
  • Data inventory listing all required systems and access methods
