Lesson 7 of 8 Stage 6: Evaluate ~20 min

Evaluate and Iterate

Your agent is built. Now you need to prove it works. Testing agentic workflows requires more than “does it run?” — you need structured test cases, clear expected outputs, and a disciplined iteration cycle.

7 / 8
1

Requirements-Based Testing

~5 min

You have a running agent. The temptation is to throw random inputs at it and see what happens. Resist that. Random testing finds random bugs. Structured testing finds the bugs that matter.

Key Principle

You wrote the scope document. You know what “correct” looks like. That makes you the right person to write test cases.

Every scope document implicitly defines five types of test cases. Each one probes a different failure mode. You need all five to have confidence that your agent is production-ready.

1 Happy Path
Standard feature request, all data available, clear output.
Tests: basic flow works end to end
2 Edge Case
Cross-team request spanning 3+ teams.
Tests: agent handles complex dependency chains
3 Data Gap
New Jira project with no history (<5 tickets).
Tests: graceful degradation with clear uncertainty flagging
4 Ambiguous Input
Similar-but-not-duplicate request.
Tests: similarity scoring at boundary thresholds
5 Boundary Conflict
Aligns with one OKR but conflicts with another.
Tests: agent surfaces tension rather than averaging

If your agent passes all five types, you have strong evidence it is ready for real inputs. If it fails on types 2–5 but passes type 1, you have a prototype — not a production system.

2

Writing Good Test Cases

~5 min

A test case is not “try this and see what happens.” A test case is a precise specification of input, expected output, and what you are checking. For each test case, document three things:

  1. Input — the specific request or trigger. Use a real historical example wherever possible.
  2. Expected Output — what a competent BA would produce for this input.
  3. What You're Checking — the specific behaviour being tested.

The expected output is the critical part. It forces you to articulate what “correct” actually looks like before you run the test. Without it, you are just observing — not testing.

Example: Happy Path Test Case
Input Standard feature request for adding a dashboard filter. Single component, clear description, no duplicates.
Expected Complexity = Small. Alignment = Medium (Indirect to UX OKR). 1 team impacted. Clean impact summary.
Checking All data retrieval succeeds, scoring matches criteria, output is complete and professional.
Example: Boundary Conflict Test Case
Input Feature request that directly advances Revenue OKR but requires resources committed to Platform Stability OKR.
Expected Agent flags both alignments. Does NOT average them. Surfaces the conflict explicitly for committee review.
Checking Conflict detection works. Priority recommendation acknowledges tension rather than resolving it.
Tip

Use real historical examples from your organisation. They anchor your test cases in reality and make the expected outputs easier to define — because you already know what the correct answer was.

3

Diagnosing Failures

~5 min

When a test fails, resist the urge to immediately tweak the prompt. First, diagnose. The symptom tells you where to look. The cause tells you what to fix.

Symptom Likely Cause Fix
Symptom
Agent misses relevant data
Likely Cause
Tool config or API query issue
Fix
Check data retrieval step
Symptom
Applies wrong criteria
Likely Cause
Decision logic not precise enough
Fix
Refine scope, update config
Symptom
Correct but poorly formatted
Likely Cause
Output template issue
Fix
Adjust template or prompt
Symptom
Simple cases OK, complex fail
Likely Cause
Edge cases not in scope
Fix
Add edge case handling
Symptom
HIL checkpoint shows too little info
Likely Cause
Checkpoint design incomplete
Fix
Surface more working, not just conclusions

The pattern here is diagnostic, not reactive. Symptom leads to cause, cause leads to fix. Prompt tweaking without diagnosis creates a different agent — not necessarily a better one.

4

The Iteration Cycle

~3 min

Evaluation is not a single pass. It is a cycle. Each round tightens the gap between what the agent produces and what you need it to produce.

1
Run
Execute test cases across all five types. Do not skip the edge cases — they are where the real problems hide.
2
Score
Compare outputs against expected results. Mark each field as correct, partially correct, or wrong. Be precise.
3
Diagnose
For each failure, identify the root cause. Use the diagnosis table. Do not guess — trace the issue back to its source.
4
Adjust
Update the scope document, design, or build artefacts. Fix the root cause, not the symptom.
5
Re-run
Verify the fix resolved the issue. Check that previously passing tests still pass — regressions are common.
Iteration Guidance

Most agents need 2–3 iteration rounds before production-ready. Track edit size at each HIL checkpoint — consistently zero edits means you can remove that checkpoint. Consistently large edits means upstream nodes need improvement.

5

Self-Check

~2 min
You notice the agent consistently gets the complexity tier right but the strategic alignment wrong. Where is the problem most likely?

The alignment assessment node. Check: is the OKR data being retrieved correctly? Are the alignment criteria in the prompt matching your scope document's taxonomy (Direct / Indirect / Neutral / Conflicting)?

If the data retrieval is fine, the issue is in the decision logic — the prompt is not specific enough about what distinguishes “Direct” from “Indirect” alignment.

After three iteration rounds, you still make significant edits at the first HIL checkpoint. What does this tell you?

The assessment nodes (complexity, alignment, team impact) need fundamental improvement. Consider: are the prompts specific enough? Is the data quality sufficient? Should you revise the scope document's decision criteria?

Three rounds without convergence suggests the problem is structural, not superficial. You may need to go back to the scope or design stage rather than continuing to iterate on the build.

What you should have after this lesson

  • 5 test cases covering all standard types
  • Expected outputs for each test case
  • Understanding of failure diagnosis and iteration
