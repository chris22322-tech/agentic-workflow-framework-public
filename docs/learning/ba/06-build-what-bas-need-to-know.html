Build — What BAs Need to Know

The build stage is where your scope and design become a working agent. Your role depends on which path your team takes — but validation is always your responsibility.

6 / 8
1

Two Paths Through Build

~5 min

The framework supports two build paths. Which one you follow depends on your team's technical capacity and the complexity of your workflow. Both produce a working agent — but your role in each is fundamentally different.

Path A

Code-first framework (Engineer-Led)

The engineer writes the code. Your scope and design documents are the specification they build from.

  • Answer edge-case questions as the engineer encounters scenarios your scope didn't cover explicitly
  • Review config — thresholds, scoring weights, and decision criteria should be configuration, not hard-coded values
  • Test early — ask to see each node working individually, not just the completed agent
Key Principle

Don't wait for the full build. Ask to see each node working individually.

Path B

Low-code builder (Self-Service)

No code required — you configure the agent directly in the builder’s UI.

  • Write the instructions text — translate your scope document's decision logic into natural language
  • Configure tools — connect to the data sources in your inventory
  • Set up approval steps — your HIL checkpoints become pause points
  • Test in the console — run your test cases directly

The rest of this lesson covers the low-code self-service path in detail, then the validation process that applies to both paths.

2

The Low-Code Build Process

~7 min

If you are following the low-code path, the build is four steps. Each one maps directly to something you already documented in your scope and design. Exact click paths vary by platform; the sequence below is the same everywhere.

1 Write the Instructions

Your scope document's decision logic gets translated into natural language instructions. This is the core of the agent — it tells the agent what to do, step by step, and what criteria to apply at each decision point.

When a feature request arrives, first search the backlog for duplicates using keyword matching. Flag any match with similarity above 0.80 as "likely duplicate". If no duplicates are found, classify the request by complexity: Simple (1-2 components affected), Medium (3-4 components), or Complex (5+ components).

The instructions text is not creative writing. It is your documented decision logic, restated as direct instructions to the agent. If your scope document was thorough, this step is mostly copy-and-translate.

2 Configure Tools

Each data source in your scope document's inventory becomes a tool the agent can call. Connect to Jira, Confluence, roadmap tools, and other systems via OpenAPI specs or Cloud Functions.

For each tool, you define what it does and what parameters it accepts. The agent uses the instructions text to decide when to call each tool and what to pass it.

If a data source does not have an API, this is where that limitation surfaces. You may need a Cloud Function to bridge the gap, or you may need to adjust the workflow to handle that data manually.

3 Set Up Approval Steps

Your HIL checkpoints from the design stage become approval steps in a low-code builder. At each approval step, the agent pauses, shows its work, and waits for human approval before continuing.

This is where your design work pays off. If you defined clear checkpoints with specific information requirements, the approval steps almost configure themselves. If your checkpoints were vague, you will struggle here.

For each approval step, define: what the agent must show (its reasoning, the data it used, its recommendation) and what actions the reviewer can take (approve, reject, modify).

4 Test in the Console

Run the test cases from your scope document against the agent directly in the console. For each test case, review the output at every approval step.

Look for: Does the agent call the right tools in the right order? Does it apply your decision criteria correctly? Does the approval step surface all the information the reviewer needs? Does it handle your documented error paths?

Do not just test the happy path. Run your edge cases. The scope document should have included these — if it did not, write them now before declaring the build complete.

Key Insight

Your scope document is 80% of what you need. The instructions text is essentially your decision logic written as instructions to the agent. If your scope was thorough, the low-code build is mostly configuration — not creation.

3

Validating the Build

~5 min

Whether you followed the code-first path or the low-code path, validation is the same. These five checks confirm the built agent matches what you scoped and designed.

1

Does the agent handle each step in your workflow map? Walk through the map node by node and confirm coverage.

2

Does it apply the decision criteria you specified — not approximate ones? Check thresholds, scoring weights, and classification rules against your scope document.

3

Do HIL checkpoints show all the information the reviewer needs to make a decision? Not just the conclusion — the reasoning and source data too.

4

Does it handle the error paths you documented? What happens when data is missing, an API call fails, or the input is ambiguous?

5

Is the output format what downstream consumers expect? If the next step is a Jira ticket, the output should be structured for Jira — not a paragraph of prose.

Validation Principle

The scope document is your acceptance criteria. If the build matches the scope, it passes. If it doesn't, the build needs to change — or the scope needs to be updated with a documented reason.

4

Common Build Pitfalls

~3 min

Three patterns that cause problems during build. Watch for each one and address them early.

Config Drift

Decision criteria from your scope end up hard-coded in the prompt or in the code instead of externalised configuration. When this happens, changing a threshold means changing the code — which means another development cycle instead of a config update.

What to do: Ask the engineer to show you where your thresholds live. If the answer is "in the prompt" or "in the code," push for them to be extracted into configuration that can be changed without a code deployment. On the low-code path, keep decision criteria as clearly labelled parameters at the top of your instructions text.

Missing HIL Context

The checkpoint shows a score or a classification but not the reasoning behind it. The reviewer sees "Complexity: High" but not why the agent concluded it was high. This forces the reviewer to either trust the agent blindly or re-do the analysis manually — defeating the purpose of the automation.

What to do: Push for the agent to surface its working, not just its conclusion. At every HIL checkpoint, the reviewer should see: the input data, the criteria applied, the reasoning chain, and the recommendation. If any of these are missing, the checkpoint is incomplete.

Scope-Design Mismatch

The built agent behaves differently from what you scoped. This is the most common pitfall and the hardest to diagnose because the gap could be in two places: the design translation (scope to design was inaccurate) or the build implementation (design to code was inaccurate).

What to do: Trace back. Run the same input through your scope document manually and compare the expected output to what the agent produces. If your manual trace matches the scope but the agent diverges, the issue is in the build. If your manual trace also diverges, the issue was in the design translation. Fix at the root, not the symptom.

5

Self-Check

On the low-code track, where does your scope document's decision logic end up?
Answer

In the instructions text — natural language instructions that tell the agent exactly what criteria to apply at each step. The instructions text is your decision logic restated as direct commands to the agent.

An engineer shows you the built agent and says "the complexity threshold is set to 3 components." But your scope document says the threshold is 4+. What do you do?
Answer

Flag it. The scope document is the source of truth. The config should match your documented criteria exactly. If there is a reason to change the threshold, that decision should be documented as a scope update — not silently changed during build.

What you should have after this lesson

  • Understanding of the two build paths (code-first and low-code) and your role in each
  • A validation checklist for reviewing the built agent against your scope
  • Awareness of the low-code self-service option and its four configuration steps
