What is an agentic workflow
and how does it map to code?

You already think in graphs, state machines, and pipeline stages. This framework gives you a structured way to scope which engineering workflows belong inside an agent—and which parts stay with you.

⏲ ~20 minutes
💻 Python fundamentals assumed
☑ Ends with a self-check
Section 1 5 min read

You already think in structured processes

Think about what you did last week. You probably triaged a batch of dependency update PRs, compiled release notes from merged pull requests, reconstructed an incident timeline from PagerDuty alerts and deploy logs, or reviewed a CI pipeline that kept failing on the same flaky test. Same pattern each time. Same steps, different data.

Now think about how much of that was genuinely hard. Not tedious—hard. The part that required understanding the codebase, knowing which changes are high-risk, judging whether a dependency bump introduces a subtle breaking change. If you are honest, the hard part was 20–30% of the total time. The rest was querying APIs, cross-referencing systems, and formatting output.

This framework helps you figure out which of those workflows an agent could handle—and which parts require your engineering judgement. It uses the same structured thinking you apply to system design, but pointed at your own work processes.

It is not a tool or a product. It is a methodology: a structured approach for decomposing your work into discrete steps, identifying what is repeatable and API-driven, and designing a graph where an agent handles the mechanical nodes while you handle the nodes that require reasoning.

Six stages. You own all of them.

The framework has six stages. They go from understanding your work all the way through to a working, tested agent. Unlike the BA pathway where engineers handle the build, you own the entire pipeline. Stages 1–3 are scoping. Stage 4 is architecture. Stage 5 is implementation. Stage 6 is evaluation.

Stages 1–3 produce a scope document—the spec for what to build. Stage 4 translates that into an agent design—actions, flow, memory shape, checkpoints. Stage 5 is implementation on whichever framework your team uses. Stage 6 is structured evaluation—trace analysis, action-level testing, iteration loops.

By the end of this learning pathway, you will go from a raw engineering workflow to a working agent design ready to implement on your framework of choice. The scope document from Stage 3 is not a handoff artifact here—it is your build spec. Every action you implement in Stage 5 traces back to a step you mapped in Stage 3.

How is this different from just building an agent directly?

You could open a file, import your framework of choice, and start coding. Most engineers do. The problem comes two weeks later when your agent handles 80% of cases but fails on the edge cases you did not think about—the PR with no labels, the release with zero changes, the rate-limited API call at 3am. Stages 1–3 force you to enumerate every step, every decision point, every error path before you write a line of code. The scope document is your test plan, your error handling spec, and your architecture sketch in one artifact.

Do I need prior agent-framework experience?

No. You need software engineering fundamentals and basic familiarity with APIs. The lessons in this pathway are framework methodology—they work regardless of the implementation technology. Once you've completed the methodology lessons, read the framework’s [Stage 4: Design](../../stages/04-design.md) and [Stage 5: Build](../../stages/05-build.md) pages and then consult your chosen platform’s documentation for implementation specifics.

Section 2 7 min read

Agents are graphs, not chatbots

Strip away the buzzwords. Most agents in code-first frameworks are a directed graph (or similar structure) with typed state. Each node (or action) is a function that reads state, does work (API call, language-model inference, data transformation), and writes results back to state. Edges connect nodes. Conditional routing decisions happen based on state values. The agent runs until it reaches a terminal state or hits a human-in-the-loop checkpoint.

If you have ever built a CI/CD pipeline, you already understand the mental model. A CI/CD workflow is a DAG: jobs are nodes, dependencies define edges, conditionals define routing, and artifacts passed between jobs are state. An agent is the same pattern, but the nodes can call language models, the state is a typed object, and the flow can pause for human review.

Key concept

An agentic workflow is not a chatbot. You are not sending prompts and parsing responses. You are defining a structure of nodes (functions), edges (transitions), typed state, and optional human-in-the-loop checkpoints. If you can draw it as a state machine, you can build it in most agent frameworks.

Notice the pause checkpoint. Most code-first agent frameworks have a primitive for this—the agent pauses execution and waits for your input. The structure in pseudocode

This is conceptual. It helps to see the mapping between the visual and the underlying structure — your chosen framework will have its own syntax for each piece:

// Memory: the fields that flow between actions memory: current_tag (string) previous_tag (string) merged_prs (list of PR records) categorised_prs (map of category to PR list) draft_notes (string) approved (boolean) // Actions: functions that read and write memory action fetch_prs: queries source control, writes memory.merged_prs action categorise: reads memory.merged_prs, writes memory.categorised_prs action generate_notes: reads categorised_prs, writes memory.draft_notes action publish: reads draft_notes, writes to release channel // Flow: the sequence between actions start -> fetch_prs -> categorise -> generate_notes -> [HIL checkpoint] -> publish // Human-in-the-loop: pause for review before publishing HIL checkpoint pauses execution and waits for reviewer response

Four actions. One HIL checkpoint. The scope document from Stage 3 maps each step in the workflow to an action, each transition to a flow edge, and each human review to a checkpoint. That is the through-line of this entire framework: scope documents become agent designs, regardless of which framework you implement them on.

What is "state" in agent-framework terms?

State is a typed object that gets passed to every action in the flow. Each action reads the fields it needs, does its work, and returns a partial update. The framework merges the update into the running state and passes it to the next action. Think of it like a context object in middleware, or the artifact that flows between CI/CD jobs. The type annotations are not decorative—frameworks use them for serialisation, checkpointing, and pause/resume.

Why not just write a plain script?

You could, and for simple workflows a script is fine. Agent frameworks pay off when you need three things: (1) human-in-the-loop checkpoints—the agent can pause, persist state, wait hours for your review, and resume exactly where it stopped. A plain script cannot do this without you building your own persistence layer. (2) Conditional routing—if the release has breaking changes, generate a migration guide; otherwise, skip that action. Most frameworks make this declarative. (3) Observability—frameworks trace every action invocation, every state mutation, every model call. When an action produces wrong output, you inspect the trace instead of adding print statements.

How does this relate to CI/CD pipelines I already build?

The mental model is almost identical. A CI/CD workflow has jobs (actions), dependencies (edges), conditional routing, outputs (state), and manual approval gates (checkpoints). The key differences: agent actions can call language models natively, the state is a rich typed object instead of flat strings, and the framework handles checkpointing and resume for you. If you squint, an agent is a CI/CD workflow where some of the jobs use a language model instead of a shell command.

Section 3 6 min read

What changes about the SWE role—and what does not

Let us address the question directly:

“If an agent can compile release notes, triage dependency updates, and reconstruct incident timelines, what is left for me?”

Everything that makes those outputs correct. An agent can fetch every PR merged since the last tag and categorise them by labels. It cannot tell you that the PR labelled enhancement is actually a bug fix that the author mislabelled. An agent can flag a dependency bump as major and check the changelog for breaking changes. It cannot tell you that the breaking change listed in the changelog does not affect your codebase because you never used the deprecated API. An agent can pull PagerDuty alerts and deploy events into a timeline. It cannot tell you that the alert at 02:47 was a false positive caused by the monitoring config change you deployed an hour earlier.

Who does what

The agent handles
  • Querying GitHub APIs for PRs, commits, labels, issues
  • Resolving squash/rebase merge commits to their PRs
  • Categorising PRs by labels and conventional commit prefixes
  • LLM-classifying PRs that lack labels or prefixes
  • Generating user-facing summaries from PR metadata
  • Formatting structured Markdown with linked references
You handle
  • Fixing mislabelled or miscategorised PRs
  • Rewriting vague or overly technical summaries
  • Catching breaking changes the automation missed
  • Validating migration notes are actually correct
  • Deciding timing and channels for the announcement
  • Adding context from conversations and tribal knowledge
The shift

Your value moves from assembling the output to validating and improving the output. The engineer who reviews an agent-drafted release notes document in 20 minutes delivers more value than the engineer who spent 3 hours building it from scratch—because the review time was spent on correctness, not on API queries and Markdown formatting.

The scope document is your architecture spec

Here is what most engineers miss when they start building agents: the scope document you produce in Stage 3 is not just planning overhead. It is the most precise description of the workflow that has ever been written down. Every step, every decision criterion (with fallback chains), every error path, every data source. When you move to Stage 4 (Design), you are not guessing at the architecture. You are translating—steps become actions, decisions become conditional routing, error paths become retry logic, and human checkpoints become pause-and-resume points.

A good scope document eliminates the most common failure mode in agent development: building an agent that handles the happy path but breaks on the edge cases you did not enumerate. Stage 3 forces you to enumerate them. Every error path in the scope document becomes error handling in your agent. Every decision criterion becomes routing logic or a model prompt.

Section 4 2 min

Quick self-check

Before you move on, take two minutes to think about your own work. These questions set you up for Lesson 2, where you will decompose your engineering role into a structured workflow inventory.

1

Think of a recurring engineering workflow you did last week that followed roughly the same steps as the last time you did it. What was it?

Release notes? Dependency updates? PR triage? Incident timeline? On-call handoff? Just picture the steps you walked through.

2

How much of that workflow operated on metadata (PR labels, API responses, timestamps) versus meaning (is this change risky, is this summary accurate, is this the right priority)?

Metadata operations are agent nodes. Meaning operations are your review at the interrupt. The more metadata-heavy the workflow, the more an agent can handle.

3

Which APIs and systems did you query? GitHub, Jira, PagerDuty, Datadog, Slack, CI/CD?

Each API is a data source the agent can query automatically. Each system you opened manually is a node you can automate. Count them.

4

Could you draw that workflow as a flow right now—actions for each step, transitions between them, and one or two places where you would want a human-in-the-loop pause?

If yes, you are already thinking like an agent designer. If not, Lesson 2 will teach you how to break any workflow into that structure.

If you can picture that workflow and feel the weight of its repetitive, API-querying, format-and-assemble middle section, you are ready for Lesson 2. That is where you will learn the four-angle decomposition method applied to engineering workflows—CI/CD, code review, incident response, release management, and more.
