Scope: Mapping Technical Workflows

You selected Release Notes Compilation as your best automation candidate. Now you map every step of how it actually works—inputs, outputs, decision logic, data sources, and the line between what the agent handles and what stays with you.

Learning Objectives
  • Map a complete engineering workflow step by step, with one action per step
  • Identify the data sources and API boundaries each step depends on
  • Document decision logic: rule-based categorisation, LLM fallbacks, and human judgement

Section 1 5 min read

From “good candidate” to build-ready spec

In Lesson 3, you scored Release Notes Compilation as the strongest automation candidate. High repeatability, low complexity, structured data throughout. But “this workflow is automatable” is not the same as “here is exactly what to build.” Scoping bridges that gap.

A scope document is a step-by-step map of how the workflow actually works today, annotated with enough detail that an engineer (or your future self in Stage 5) can translate it directly into a graph architecture. Every step has one action, one input, one output, and a boundary tag that says who handles it.

The scope document is your build spec. If the engineer has to guess how a step works, the scope is incomplete. If a step hides two different operations, it needs to be split. One step, one action, one boundary tag.

How is this different from a technical design doc?

A design doc describes how the agent will be implemented—graph nodes, state schema, conditional edges. A scope document describes how the workflow works today as a human performs it. The scope comes first. It captures the domain logic, decision criteria, and data dependencies that the design then translates into architecture. If you skip the scope and go straight to design, you are encoding assumptions about the workflow instead of documenting what actually happens. Stage 4 (Design) takes the scope document as its input.

Why one action per step?

Because each step gets exactly one boundary tag: AUTOMATE, HIL, or MANUAL. If a step contains both “fetch PRs from GitHub” (AUTOMATE) and “decide which PRs are user-facing” (also AUTOMATE, but different decision logic), you cannot assign a single boundary tag without hiding complexity. Worse, during Design, each step maps to a node or part of a node in the agent graph. A step that does two things becomes a node that does two things—harder to test, harder to debug, harder to iterate on.

Nine automated steps, one human-in-the-loop review, one manual publication. That ratio—9 AUTOMATE, 1 HIL, 1 MANUAL—is what made this workflow score so well in Stage 2. Let us walk through three key steps in detail to see what good scoping looks like.

Step 2: Fetch merged PRs since last release

This step sounds simple: “get the PRs.” But scoping forces you to document exactly how:

  1. Input: The current tag and previous tag from Step 1 (e.g., v2.4.0 and v2.3.0)
  2. Action: Call the GitHub Compare API (/repos/{owner}/{repo}/compare/{base}...{head}) to get the commit range, then resolve each merge commit to its PR via /commits/{sha}/pulls
  3. Edge case — squash merges: One commit maps directly to one PR. Clean.
  4. Edge case — rebase merges: Each rebased commit is separate. Must query per-commit and deduplicate by PR number to avoid listing the same PR twice.
  5. Output: List of merged PRs, each with: PR number, title, author, labels, merge commit SHA, branch name, description body, merge timestamp

Notice the merge strategy edge case. If you wrote “fetch PRs between two tags,” you would miss the fact that rebase-merged PRs require per-commit resolution and deduplication. The engineer building this in Stage 5 needs to know that, or the agent will list the same PR three times for a rebased branch with three commits.

Step 4: Categorise changes by type

This is the most complex automated step. Not because it is hard to implement, but because it has four classification strategies applied in priority order:

Strategy 1 — Rule
Label match

Map PR labels to categories: feature/enhancement → New Features, bug/bugfix → Bug Fixes, security → Security, breaking-change → Breaking Changes. If multiple labels match, prefer the highest-severity category.

Strategy 2 — Rule
Conventional commit prefix

If no label matched, parse the PR title for a conventional commit prefix: feat: → New Features, fix: → Bug Fixes, perf: → Performance, docs: → Documentation, chore:/ci: → Internal.

Strategy 3 — LLM
LLM classification

If neither label nor prefix matched, send the PR title, description (first 500 characters), and linked issue types to the LLM: “Classify this PR into exactly one category. If ambiguous, classify as Other and flag for manual review.” Log every LLM-classified PR.

Strategy 4 — Rule
Breaking change override

Regardless of primary category, if any commit message contains BREAKING: or BREAKING CHANGE:, add the PR to Breaking Changes as well. A PR can appear in both its primary category and Breaking Changes.

Why this matters for scoping

Without the scope document, an engineer might implement a single classification call to the LLM for every PR. That works, but it is slower, more expensive, and less deterministic than checking labels first. The priority chain—labels → commit prefixes → LLM—is a design decision that must be captured in the scope, because it comes from understanding how the team actually works, not from technical architecture.

Why is Step 4 tagged AUTOMATE if it uses an LLM?

Because categorisation here is a flagging step, not a deciding step. The agent proposes a category; the engineer validates it in Step 10. A miscategorised PR appears in the wrong section of the draft—that is caught in seconds during review. The cost of a wrong classification is a minor correction, not a wrong decision shipped to users. The boundary would only move to HIL if categorisation errors had consequences beyond “the reviewer fixes it.”

Step 10: Review and refine (the single HIL checkpoint)

This is the only human-in-the-loop step in the entire workflow. The engineer validates six dimensions:

  1. Categorisation accuracy — Any PRs in the wrong section? Common error: an enhancement-labelled PR that is actually a bug fix.
  2. Summary quality — Do the one-liners accurately describe the change for users? LLM summaries may be too vague (“Improved performance”) or too technical.
  3. Excluded changes — Review the exclusion list for false exclusions: internal-labelled PRs with user-facing side effects.
  4. Breaking change completeness — Is every breaking change accounted for? Are the migration notes accurate?
  5. Ordering and emphasis — Does the section order reflect what users care about? Move the headline feature to the top.
  6. Tone and audience fit — Internal platform notes differ from public API notes.

One HIL checkpoint instead of three scattered ones. This works because errors in the release notes workflow do not compound—a miscategorised PR does not cause other PRs to be miscategorised. Each error is independent and catchable in a single review pass.

Section 3 6 min read

Decision logic: the part most people skip

Every step with non-trivial logic gets a Decision Logic entry in the scope document. This is where you capture the rules—not just “categorise the PR,” but the exact criteria, the priority order, and the fallback chain. The engineer should be able to implement the step from this column alone.

Three types of decision logic in engineering workflows

Type 1
Rule-based (deterministic)

Label bug → category Bug Fixes. Label no-release-notes → exclude from output. PR with >10 files changed → sort higher. These are lookup tables and threshold checks. Zero ambiguity, zero LLM involvement. Most of Steps 4, 5, and 7 are rule-based.

Type 2
LLM-assisted (probabilistic with logging)

When rules cannot decide, the LLM fills the gap. Step 4 uses it for unlabelled PRs. Step 6 uses it for vague titles. Step 9 uses it for sparse migration instructions. Every LLM call is logged—which PRs required it, what the LLM returned—so the team can improve their labelling and PR templates over time.

Type 3
Human judgement (irreducible)

Step 10: is the tone right for the audience? Step 11: should we publish now or wait for the deploy to succeed? Should the Slack announcement emphasise the security fix? These decisions require context the agent cannot access—the political landscape, the deployment status, the audience expectations.

Breaking change detection: a known gap

Good scoping documents the things that will not work perfectly. Breaking change detection has a gap: the agent detects breaking changes via two signals—the breaking-change label and the BREAKING: commit prefix. Both require the PR author to explicitly flag the change. If the author forgets both, the agent misses it.

This is a process gap, not a solvable one at the agent level. The engineer’s review in Step 10 is the safety net. Documenting this gap in the scope document means the engineer knows it exists and can add heuristics later—like flagging PRs that modify public API route definitions or change exported function signatures.

Could the LLM detect breaking changes from diffs?

In theory, yes—you could send the diff to an LLM and ask “does this change break backward compatibility?” In practice, this crosses the metadata/meaning line from Lesson 3. The LLM would need to understand what the code does, not just read labels and titles. That is a different class of complexity, with higher token cost, higher latency, and lower reliability. The scope document draws the boundary at metadata. If the team later wants diff-level analysis, that is a v2 scope expansion—documented as a known gap, not ignored.

Data inventory: what the agent needs to read

Every automated step needs data from somewhere. The data inventory catalogues every external system the agent will talk to—the access method, the auth requirements, and the gotchas. This becomes the integration spec for Stage 4: Design.

Data Source Access Method Auth Key Gotcha
GitHub API
(PRs, commits, labels)		 REST v3 or GraphQL v4 PAT or GitHub App token Compare API returns max 250 commits. GraphQL is more efficient—one query for PR metadata, labels, and commits. Rate limit: 5,000 req/hr.

(Jira or Linear)		 REST (Jira v3) or GraphQL (Linear) API token / key Jira custom fields have system-generated IDs like customfield_10016 that vary per instance. Discover via the field metadata endpoint.
CI/CD pipeline
(GitHub Actions, CircleCI)		 REST API Same GitHub token (Actions) or separate API token Verify the release tag’s build passed before generating notes. A failed CI run should halt or flag a warning.

Git repository
(tags, commit history)		 Git CLI or GitHub API File system access or GitHub token Tag → commit SHA resolution anchors the entire PR query range. Local access is faster; remote works from any environment.
Why the data inventory matters

Each row in this table becomes an integration point in the agent graph. If a data source requires a separate API token, the agent needs a credential. If a data source has a rate limit, the agent needs retry logic. If a data source has a gotcha (Jira’s custom field IDs), the agent needs a discovery step. None of this is visible from “fetch the PRs.” It only becomes visible when you catalogue the sources.

What about data sensitivity?

The scope document also catalogues what data gets sent to the LLM. PR descriptions may contain security fix details or references to vulnerabilities. Issue tracker content may include customer-reported bugs with PII. Security-labelled PRs should bypass LLM classification and be manually categorised instead. These governance constraints are captured in the scope’s Constraints and Assumptions section—not as an afterthought, but as a prerequisite for build.

Section 5 4 min read

Boundary tagging: drawing the automation line

Every step gets exactly one boundary tag. The tag answers a single question: who executes this step?

Release Notes — Automation Boundary
Agent handles (AUTOMATE)
  • Step 1: Detect release tag event
  • Step 2: Fetch merged PRs via GitHub API
  • Step 3: Resolve linked issues from tracker
  • Step 4: Categorise by labels, prefixes, LLM
  • Step 5: Filter to user-facing changes
  • Step 6: Extract one-line change summaries
  • Step 7: Group by category, sort by impact
  • Step 8: Generate formatted Markdown draft
  • Step 9: Generate migration notes (if breaking)
Boundary
You handle (HIL + MANUAL)
  • Step 10: Validate categorisation accuracy
  • Step 10: Rewrite vague summaries
  • Step 10: Check excluded list for false exclusions
  • Step 10: Verify breaking changes are complete
  • Step 10: Adjust ordering and emphasis
  • Step 10: Confirm tone fits the audience
  • Step 11: Decide when and where to publish
  • Step 11: Frame the announcement

Why only one HIL checkpoint?

Other workflows in the framework have three or four HIL checkpoints. Release notes has one. The reason is error independence. In a workflow where a wrong assessment in Step 5 cascades through Steps 6, 7, and 8, you need early checkpoints to prevent compounding. In release notes, a miscategorised PR in Step 4 does not cause other PRs to be miscategorised. Each PR’s classification, summary, and position are independent. All errors can be caught in a single review pass.

The boundary should err toward more human involvement in early versions. If the team’s labelling discipline is poor and the LLM fallback produces frequent errors, add a checkpoint after Step 4—“review the categorisation before generating summaries.” Start with a tighter boundary and relax it as you gain confidence.

When would Step 11 (Publish) become AUTOMATE?

When the team adopts a convention where every release is published immediately after approval, and the distribution channels are always the same (GitHub Release + CHANGELOG.md commit). At that point, publishing becomes a deterministic API call with no judgement involved. But start MANUAL until the team confirms the publishing decision is always the same. Premature or misdirected publication is difficult to reverse.

Section 6 3 min

Self-check

Before moving on to the framework’s design and build methodology pages, check your understanding of scope documents. These questions test whether you could produce one for your own workflow.

1

Step 4 uses an LLM for ambiguous PRs, yet it is tagged AUTOMATE. What is the reasoning?

Think about the cost of a wrong classification. Where is the error caught? What would change if the error could not be caught later?

2

The scope document lists 5 data sources. Pick any one and explain what happens to the workflow if that source is unavailable.

GitHub API rate-limited? Jira returning errors? The answer reveals whether you understand the error path register—which downstream steps are affected and how the workflow degrades.

3

Why does the release notes workflow have one HIL checkpoint while other workflows in the framework have three or four?

The answer is not “it is simpler.” Think about how errors propagate. Do classification errors in Step 4 affect summary quality in Step 6?

4

Breaking change detection has a known gap. Why does the scope document call it out explicitly instead of trying to solve it?

Think about the difference between a process gap and a technical limitation. Where is the safety net?

🔑 Key Takeaway

A scope document is precise enough that the engineer never has to guess. Every step has one action, one input, one output, and one boundary tag. The decision logic captures the rules, the priority order, and the fallback chain. The data inventory catalogues every integration point with its auth requirements and gotchas. The error path register documents what breaks and how the workflow degrades.

The scope document you produce in Stage 3 is the single most important artifact in the framework. It is the bridge between “this workflow is automatable” and “here is exactly what to build.” If you get the scope right, Design (Stage 4) is translation. If you get it wrong, the engineer builds what they think you do instead of what you actually do.
