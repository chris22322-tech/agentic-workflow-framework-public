Select: Scoring Engineering Candidates

Section 1 4 min read

Why you cannot just pick the obvious one

After decomposing your role in Lesson 2, you have a table of 10–12 engineering workflows with automation potential ratings. The temptation is to pick the one that sounds most impressive (“let’s automate incident response!”) or the one that annoys you most (“I hate writing release notes”). Both are bad selection criteria.

Impressive-sounding workflows are often impressive because they are hard—they involve judgement, ambiguity, and context that changes every time. The workflow that annoys you might annoy you precisely because it is boring and repetitive, which actually makes it a great automation candidate. But you need a structured way to distinguish between “boring and automatable” and “boring but full of hidden complexity.”

The six-dimension scoring framework replaces intuition with structured comparison. Each dimension isolates a different risk factor: will it save time? Can the agent access the data? What happens when it gets something wrong?

The framework uses six dimensions, each weighted differently. The weights reflect what matters most for a first build—your initial agent, where success matters more than ambition because it sets the precedent for every future automation project on your team.

Why is the first build special?

Your first agent is not just a productivity tool. It is a proof of concept. If it works well, your team gains confidence in the approach and you build political capital for the next one. If it fails—or worse, produces output that makes someone look bad—you are fighting an uphill battle for every future project. The scoring weights are calibrated for this reality: Impact and Feasibility are weighted highest because the first build needs to both save real time and actually work.

Can I change the weights for my context?

Yes, but carefully. The default weights (Impact 30%, Feasibility 25%, Risk Tolerance 20%, Complexity 10%, Learning Value 5%, Org Readiness 10%) are calibrated for a first build. If you are building your third or fourth agent and your team already trusts the approach, you might increase Learning Value weight to prioritise patterns that transfer to future builds. But do not change weights to justify a decision you have already made—that defeats the purpose.

Section 2 5 min read

Six dimensions, through an engineering lens

The scoring dimensions are the same across all roles, but how they manifest in engineering workflows is specific. Here is each dimension with the engineering-specific question you should be asking.

📈
Impact
Weight: 30%
“How many engineer-hours per month does this reclaim?”
Time per occurrence × frequency × number of instances. A 4-hour workflow that runs biweekly for 1 service is different from a 30-minute workflow that runs 80 times a month. Calculate the actual number.
🔌
Feasibility
Weight: 25%
“Does every data source have an API?”
Engineering tools are built for developers—GitHub, Jira, PagerDuty, CI/CD systems all have APIs. This is your structural advantage. The question is whether any step requires interpreting unstructured data: Slack conversations, meeting notes, design docs.
🛡
Risk Tolerance
Weight: 20%
“What happens when the agent gets it wrong?”
A miscategorised release note is caught in review. A wrong reviewer assignment wastes someone’s afternoon. A botched incident escalation pages the wrong team at 3am. The failure cost varies enormously across engineering workflows.
🛠
Complexity
Weight: 10%
“Is the hardest step metadata or meaning?”
From Lesson 2: workflows that operate on metadata (PR labels, version numbers, alert timestamps) score higher than workflows that interpret meaning (is this change risky? which debt is urgent?). Linear flows with deterministic rules score 5. Flows with judgement-dependent branching score 2–3.
📚
Learning Value
Weight: 5%
“Does the pattern transfer to other workflows?”
A “data extraction → classification → formatted output” pattern appears in changelogs, deployment summaries, dependency reports, and compliance documentation. Build one, and you have the template for the next four.
🏢
Organisational Readiness
Weight: 10%
“Will the team actually use it?”
Some engineers are protective of their processes. If the senior engineer who “owns” code review views automated triage as undermining their role, you will fight adoption regardless of the agent’s quality. Pick a workflow where the team actively wants relief.
The engineering advantage

Engineering workflows have a structural advantage on Feasibility that most other roles do not: your tools are built by developers, for developers. GitHub, Jira, Linear, PagerDuty, Datadog, CI/CD pipelines—all have well-documented APIs with structured JSON responses. When a CSM scores Feasibility, they worry about whether Slack sentiment analysis is reliable. When you score Feasibility, the question is narrower: does any step require interpreting unstructured human communication? If the answer is no, you are likely starting at Feasibility 4–5.

Reading the comparison table

Three patterns emerge from this table that apply to engineering workflows generally:

Release Notes wins on breadth, not on any single dimension

Release Notes Compilation does not dominate every column. On-Call Rotation Handoff ties it on Feasibility (5) and beats it on Risk Tolerance (5 vs 4). PR Review Triage ties it on Impact (4). But Release Notes has no score below 4. It wins the composite because it has the strongest overall profile—no dimension drags it down.

On-Call Handoff is pragmatic but modest

On-Call Rotation Handoff has the best pragmatism profile on the table: Feasibility 5, Risk Tolerance 5. Every system has an API, and the handoff document is reviewed before the incoming engineer relies on it. But Impact scores only 3: at 1–2 hours weekly, it saves roughly 4–6 hours per month with no instance multiplier. Strong on “can we build it safely?”—modest on “is it worth building?”

PR Review Triage: highest frequency, lowest composite

PR Review Triage runs 3–5 times per day—by far the highest frequency on the table. The intuition is that high frequency equals high impact. And Impact does score 4. But the composite is dragged down by Complexity (2) and Risk Tolerance (3). The data gathering is straightforward, but the decision—who reviews, how urgently, is this blocking a release—requires engineering context that lives in people’s heads. Frequency is an Impact multiplier, not a success predictor.

The SE selection lesson

The metadata vs meaning distinction from Lesson 2 is the strongest predictor of composite score. Release Notes operates on metadata—PR labels, commit messages, version tags. PR Review Triage operates on meaning—“is this change risky?” requires understanding what the code does. On-Call Handoff sits between: data gathering is metadata, but the valuable parts (what to watch for, which alerts to ignore) are meaning. If your candidate’s hardest step is about retrieving and reformatting structured data, it will likely score well. If it is about interpreting what that data means in context, it will not.

Why Feasibility scored 5 — the engineering structural advantage

Engineering tools are built by and for developers, which means API access is a first-class feature. GitHub’s REST and GraphQL APIs expose everything: PR metadata, commit history, labels, milestones, linked issues. Jira and Linear have well-documented REST and GraphQL APIs. CI/CD systems expose build and test results via API.

Compare this to a Customer Success Manager scoring Feasibility, where Slack sentiment analysis was the weak link—retrieving threads is easy, but interpreting conversational tone is inherently noisy. For engineering workflows, if your data sources are Git, CI/CD, monitoring, and issue tracking, you are likely starting at Feasibility 4–5. The question is whether any step requires interpreting unstructured data. If it does, that is what drops the score.

Why Complexity scored 5 — the metadata advantage

Release notes categorisation asks: “Does this PR have the bugfix label? Then it is a bugfix.” That is a metadata operation. PR review triage asks: “Is this 10-line change to the payment service risky enough to need a senior reviewer?” That is a meaning operation—it requires understanding what the change does, not just what labels it carries.

Metadata operations are deterministic and encodable as rules. Meaning operations require context and judgement. When your workflow’s hardest step is a metadata operation, Complexity stays high.

What are the risks the scoring surfaces?

Two risks are worth flagging. First, categorisation accuracy depends on PR labelling discipline. If the team does not consistently label PRs, the agent falls back to commit message parsing, which is noisier. Expect to iterate on the categorisation prompt for edge cases. Second, breaking change detection has a gap. The agent detects breaking changes via labels and commit prefixes. If the author forgets both signals, the agent will miss it. This is a process gap, not an agent gap—the engineer’s review step is the safety net.

Section 5 2 min

Self-check

Before you move to Lesson 4 (Scope), make sure these concepts have landed. Think about your own workflow inventory from Lesson 2 as you answer.

1

Pick your top three candidates from Lesson 2. For each one, is the hardest step a metadata operation or a meaning operation?

If you are unsure, ask: “Would two engineers with the same inputs produce the same output for this step?” If yes, it is metadata. If no, it is meaning.

2

Which of your candidates has the highest Feasibility? Does every data source have an API, or does any step require interpreting Slack messages, meeting notes, or other unstructured text?

Remember: engineering tools almost always have APIs. The question is whether the workflow also touches unstructured human communication.

3

What happens when each candidate’s agent gets something wrong? Is the error caught in a review step before it affects anyone, or does it go straight to production / external users / on-call engineers?

This is Risk Tolerance. A review-before-publish checkpoint is the most forgiving failure mode. Direct-to-production is the least.

4

Calculate the actual monthly hours for your top candidate: time per occurrence × frequency × number of instances. Is it above 6 hours/month?

If the number is below 4 hours/month, consider whether the automation is worth the build effort for a first project. Small time savings can still be worth it if Feasibility and Complexity are both high.

🔑 Key Takeaway

The metadata vs meaning distinction is the strongest predictor of automation success for engineering workflows. Release Notes Compilation won not because it saves the most time or sounds the most impressive, but because every step operates on structured data with deterministic rules. The agent retrieves PR labels, applies classification logic, and produces formatted output. No step requires the agent to understand what code changes mean.

When evaluating your own candidates, ask this for each workflow’s hardest step: is it about retrieving and reformatting structured data, or about interpreting what that data means in context? The answer will often predict the composite score before you calculate it.
