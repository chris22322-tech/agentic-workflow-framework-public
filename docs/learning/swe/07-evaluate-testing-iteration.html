Evaluate: testing and iteration

Your agent runs. Now you need to prove it works—with the same rigour you bring to any production system. Unit tests for tools, integration tests for the graph, structured diagnosis when things fail, and a disciplined iteration cycle.

⏲ ~25 minutes
🔧 Structured agent testing patterns
☑ Ends with a self-check
Section 1 6 min read

An agent is not a black box—test it like code

You would never ship a service without tests. An agent is no different—it is a graph of functions with typed state, external API calls, and LLM invocations. Every one of those components is testable. The challenge is knowing what to test at which level and how to keep tests fast, deterministic, and useful.

Most agent failures originate in a single layer: a tool that mishandles an edge case, a prompt that misclassifies a signal, a routing function that picks the wrong branch. If you test only end-to-end, you burn API budget and time on every run, and when a test fails, you cannot tell which layer broke. The engineering approach is the same one you use for any system: unit tests for components, integration tests for the assembled system, and a clear strategy for each.

Learning objectives

By the end of this lesson, you will be able to: (1) design test cases that cover happy paths, edge cases, and failure modes for an agent; (2) use the four-layer diagnostic technique to pinpoint which layer caused a failure; (3) write unit tests for tool functions, analysis nodes, and routing logic; (4) structure an iteration cycle that fixes the right thing at the right layer.

Three levels of agent testing

Think of agent testing as three concentric rings. The inner ring is fast and cheap. The outer ring is slow and expensive. You want most of your coverage in the inner ring, with the outer ring as a safety net.

The Agent Testing Pyramid
1
Unit Tests
Fast & free
2
Integration Tests
Full graph
3
Evaluation Runs
Real scenarios

Unit tests isolate individual node functions—mock external dependencies, assert on outputs. Integration tests invoke the compiled graph end-to-end with controlled inputs. Evaluation runs compare agent output against known-good manual results.

What to test at each level

Unit tests (run every commit)
  • Tool functions: does fetch_merged_prs correctly parse the GitHub API response?
  • Analysis nodes: given a known LLM response, does the node extract the right fields?
  • Routing functions: given a state with health_score: "Red", does it route to risk_assessment?
  • Error handling: does the node set a warning flag when the API returns empty data?
Integration / regression tests (run in CI)
  • Full graph execution with known inputs and expected state fields at the end
  • HIL checkpoint behaviour: does the graph pause at the right point and resume correctly?
  • Conditional edge routing: does a release with breaking changes produce a migration guide section?
  • Regression: does a prompt change break a test case that was previously passing?

Unit tests run in milliseconds and cost nothing. They catch the majority of node-level bugs. Save the full-graph tests—which require real LLM calls—for CI runs where the cost is justified. The ratio matters: if your first instinct is to test the whole graph every time, you will iterate slowly and expensively.

Designing test cases

Before you write a single test, design the scenarios. A minimum of five test cases, each targeting a different profile. Do not just test the happy path—the edge cases are where agents fail.

Test Case Profile What It Tests
Happy path Normal input, all data available The basic flow works end to end. Every node produces expected output.
At-risk scenario Declining metrics, open critical issues The agent correctly identifies and escalates risk signals. Conditional edges fire.
Data gap One or more APIs return empty or error Graceful degradation—the agent flags the gap, does not hallucinate data.
Sparse data Limited history, few records The agent does not over-interpret sparse data or produce false confidence.
Complex case Multiple signals, mixed indicators The agent handles nuance and does not oversimplify multi-dimensional inputs.

For each test case, record: the input state, the expected behaviour at each node, the actual behaviour, a verdict (Pass / Partial / Fail), and notes explaining what went wrong or right. A Pass is not “it worked”—record anything surprising. A Fail needs enough detail that someone else could implement the fix without asking you questions.

What does a good Fail write-up look like?

Example: Release notes omit all 4 security patches. Root cause: fetch_merged_prs filters by labels ["feature", "bugfix", "enhancement"]. PRs labelled security are never retrieved. This is a tool issue—the data never reached the prompt. Fix: add "security" and "breaking-change" to the label filter, and add a Security section to the release notes template. A good Fail write-up names the root cause, identifies the layer, and describes the fix concretely enough that someone else could implement it.

When is Partial acceptable?

A Partial means the agent produces usable output that requires some human correction. If the agent handles 80% of the work and you spend 10 minutes correcting the rest, that may already be a significant time saving. The question is not “is it perfect” but “is agent-plus-correction meaningfully faster than doing it manually?” Every remaining Partial should have a known, documented root cause and the human correction should be fast and predictable.

Section 2 7 min read

The four-layer diagnostic technique

When a test case fails, the temptation is to tweak the prompt. It is the most accessible fix. But failures frequently present as one category and originate in another. A blended usage score looks like a prompt issue, but it may be a scope issue where the schema has nowhere to store per-product results. If you default to the easiest fix, you waste an iteration cycle and the failure persists.

Use systematic elimination. Check each layer in order, starting from the data and working outward. The first layer where the answer is “no” or “it structurally cannot” is the failure category. Do not skip ahead.

1
Tool layer
“Was the right data retrieved?”
Check whether the tool returned correct, complete data. If the raw data is wrong or missing, the failure is a tool issue regardless of what the other layers did. Cheapest fix, smallest blast radius.
2
Prompt layer
“Given the data it received, did the prompt produce the right analysis?”
Assume the data is correct (you just verified it). Read the prompt’s input and output. Did the LLM do what the prompt asked? Missing instructions, no weighting criteria, no edge case guidance = prompt issue.
3
Graph layer
“Is the node in the right position with the right inputs?”
Assume data and analysis are correct. Check whether the node receives inputs from the right upstream nodes, whether conditional edges route correctly, and whether HIL checkpoints are placed where they can catch problems.
4
Scope layer
“Does the scope support what the agent needs to represent?”
Assume data, analysis, and structure are all sound. Check whether the scope—the steps, their outputs, the state schema they imply—can represent the distinctions this test case requires. If the schema has no field for per-product analysis, no prompt or graph change can produce per-product output. Most expensive fix, cascades through Stages 3 → 4 → 5.
Why the order matters

Tool fixes are cheapest (change a query or API call) and have the smallest blast radius. Prompt fixes are next (change instructions, re-run the node). Graph fixes require updating the design and rebuilding affected nodes. Scope fixes cascade through three stages. By checking in this order, you find the cheapest adequate fix first and avoid unnecessary cascading changes.

The technique in action

Here are three failures that land on different layers. Notice how each one looks like a prompt issue at first glance but is not.

Tool issue: release notes omit hotfix PRs

The agent listed 18 changes; manual notes had 22. The four missing items were all hotfix PRs merged to release/2.3.x and cherry-picked to main.

Layer 1 — Tool: The fetch_merged_prs tool queries for PRs merged to main since the last release tag. Cherry-pick commits do not appear as merged PRs in the GitHub API response. Data is incomplete. Stop here.

Fix: Update the tool query to include PRs merged to release/* branches, or detect cherry-pick commits and trace them back to their original PRs.

Prompt issue: renewal risk report understates urgency

The agent rated risk as “Moderate” for an account 45 days from renewal with usage down 40% and two unresolved P1 tickets. The manual brief rated it “High Risk.”

Layer 1 — Tool: Data returned correctly—usage decline (40% QoQ), two open P1 tickets, and the renewal date. ✓

Layer 2 — Prompt: The prompt assesses risk based on usage trends and ticket volume independently. Each signal alone rates “Moderate.” The prompt has no compounding logic for multiple signals near a deadline. Stop here.

Fix: Add compounding logic: “When multiple negative signals are present simultaneously and a renewal is within 60 days, escalate the overall risk rating by one level.”

Graph issue: duplicate feature requests not caught

The agent processed a feature request nearly identical to an existing backlog item but did not flag the duplicate.

Layer 1 — Tool: The fetch_backlog tool returned the full backlog including the existing similar item. ✓

Layer 2 — Prompt: The duplicate-check prompt was never invoked. ✓ (not the issue)

Layer 3 — Graph: The duplicate-check node is placed after prioritisation. The conditional edge from prioritisation routes low-priority items directly to the backlog, bypassing duplicate checking. This request was assessed as low priority, so it skipped the check entirely. Stop here.

Fix: Move the duplicate-check node before prioritisation. Every request should be checked for duplicates regardless of priority.

The pattern: start at the data, work outward. The first layer that did not do its job correctly is where you fix. Do not default to prompt tweaking—it is the most tempting fix but frequently the wrong one.

Section 3 7 min read

Writing tests for your agent

Three patterns cover the majority of unit test cases for agent nodes. Each isolates the node function, replaces external dependencies with controlled substitutes, and asserts on the output given known input.

Pattern 1: Mock tool responses for data-gathering nodes

A data-gathering node calls an external API and writes the result into state. The logic you are testing is what the node does with the response—parsing, filtering, error handling—not whether the API itself works. Replace the API call with a fixture.

# tests/test_gather_node.py from unittest.mock import patch from your_agent.nodes import gather_data def test_gather_data_parses_usage_metrics(): """The node should extract monthly active users from the API response.""" mock_response = { "metrics": [ {"name": "monthly_active_users", "value": 1423}, {"name": "api_calls", "value": 58210}, ], "period": "Q4", } with patch("your_agent.nodes.usage_api_client.get_metrics", return_value=mock_response): state = gather_data({"account_id": "ACME-001", "period": "Q4"}) assert state["usage_data"]["monthly_active_users"] == 1423 assert state["usage_data"]["api_calls"] == 58210 def test_gather_data_handles_empty_response(): """The node should flag missing data, not crash or hallucinate.""" with patch("your_agent.nodes.usage_api_client.get_metrics", return_value={"metrics": [], "period": "Q4"}): state = gather_data({"account_id": "NEW-001", "period": "Q4"}) assert state["usage_data"] == {} assert "missing_usage_data" in state.get("warnings", [])

Pattern 2: Mock LLM responses for analysis nodes

An analysis node sends a prompt to the LLM and processes the response. The logic you are testing is the node’s handling of the LLM output—parsing JSON, extracting scores, classifying risk. Replace the LLM call with a function that returns a deterministic response.

# tests/test_analysis_node.py from unittest.mock import patch from your_agent.nodes import analyse_health def _fake_llm_response(prompt, schema=None) -> dict: """Return a deterministic LLM response for testing.""" return {"health_score": "Red", "risk_factors": ["usage_decline", "open_p1"], "confidence": 0.87} def test_analyse_health_parses_risk_factors(): state = { "usage_data": {"monthly_active_users": 300, "trend": "declining"}, "support_data": {"open_p1_tickets": 2}, } with patch("your_agent.nodes.call_llm_for_json", side_effect=_fake_llm_response): result = analyse_health(state) assert result["health_score"] == "Red" assert "usage_decline" in result["risk_factors"] def test_analyse_health_handles_malformed_output(): state = {"usage_data": {"monthly_active_users": 300}, "support_data": {}} with patch("your_agent.nodes.call_llm_for_json", return_value="not valid json"): result = analyse_health(state) assert result.get("analysis_error") is not None

Pattern 3: Test routing functions with synthetic state

Routing functions are pure logic—no API calls, no LLM calls. They take state and return the name of the next node. Construct a state dict for each branch condition and assert on the return value. No mocking needed.

# tests/test_routing.py from your_agent.routing import route_after_analysis def test_routes_to_risk_assessment_when_red(): state = {"health_score": "Red", "risk_factors": ["usage_decline"]} assert route_after_analysis(state) == "risk_assessment" def test_routes_to_report_when_green(): state = {"health_score": "Green", "risk_factors": []} assert route_after_analysis(state) == "generate_report" def test_routes_to_hil_review_when_amber(): state = {"health_score": "Amber", "risk_factors": ["open_p1"]} assert route_after_analysis(state) == "human_review"

These three patterns—mock tools, mock LLMs, test routing with synthetic state—cover the majority of node-level tests. They run in milliseconds and cost nothing. Run them on every commit.

Integration tests: invoking the full agent

Integration tests run the full agent end-to-end with controlled inputs. They use real model calls, so they are slow and cost API budget. Save them for CI. The key technique: define fixtures with input and expected values for critical memory fields, and auto-resume past HIL checkpoints with predetermined responses.

// Integration test pseudocode import your_agent.build_agent // Fixtures: input + expected output fields FIXTURES = { "happy_path": { "input": { account_id: "ACME-001", period: "Q4" }, "expected": { health_score: "Green", risk_flags: [] }, }, "at_risk": { "input": { account_id: "RISK-042", period: "Q4" }, "expected": { health_score: "Red", risk_flags: ["usage_decline", "open_p1"] }, }, } // Predetermined HIL responses for unattended execution HIL_RESPONSES = { "review_analysis": "Looks good, no corrections.", "review_report": { decision: "approved" }, } function test_regression(case_name, case): agent = build_agent(persistence: in_memory_test_store) // Initial invocation agent.run(case["input"], run_id: "test-" + case_name) // Resume through any HIL checkpoints while agent.is_paused: checkpoint_name = agent.current_checkpoint feedback = HIL_RESPONSES.get(checkpoint_name, { decision: "approved" }) agent.resume(feedback, run_id: "test-" + case_name) // Assert on critical memory fields final = agent.final_memory assert final.health_score == case["expected"].health_score
Why assert on memory fields, not full output text?

Model output varies between runs. Asserting on exact text creates flaky tests. Assert on scores, classifications, the presence of required sections, and risk flags. If you need to assert on output quality, use a model-as-judge evaluator (most modern agent platforms provide one, or you can build a small one).

Why should agent construction and runtime be separate?

If the agent construction function bakes in a fixed persistence backend, tests cannot inject their own lightweight store and production code cannot swap in a durable one. Keep agent construction and runtime configuration separate: the builder wires actions and flow, the call site configures persistence and other runtime options. Tests use an in-memory test store; production uses a durable one.

What about failure injection?

Deliberately inject failures to verify your agent degrades gracefully. Simulate rate limits, empty responses, malformed data, and timeouts. For the release notes agent: feed a release with zero PRs and verify you get “No changes in this release” rather than an empty file. Simulate a rate-limit error during fetch_prs and verify the agent waits for the reset window rather than crashing. These are unit-testable with mocks—you do not need the real API to test error handling.

The iteration cycle: when to fix what

After each evaluation round, follow a disciplined cycle. The goal is not just to fix failures but to fix them at the right layer. A prompt tweak applied to a scope issue wastes an iteration cycle. A scope rewrite applied to a tool bug wastes a week.

Fix scope determines blast radius

The failure category determines how much you touch. Verify you have the right category before starting.

Fix Category What You Change Blast Radius
Tool fix A query, API call, or data parsing function One file. Re-run test cases that use that tool.
Prompt fix LLM instructions, classification criteria, output format One prompt. Re-run test cases that invoke that node.
Graph fix Node ordering, conditional edges, HIL checkpoint placement Design document + affected node implementations. Re-run all test cases touching modified nodes.
Scope fix Workflow steps, boundary tags (AUTOMATE/HIL/MANUAL), state schema Scope → Design → Build. Cascades through three stages. Re-run all affected test cases.
Key principle

A misdiagnosed scope issue treated as a prompt fix wastes an iteration cycle. A scope fix misidentified as a graph fix wastes days. Use the four-layer technique to find the cheapest adequate fix, then verify the fix does not break other test cases before moving on.

When to stop iterating

Not every agent needs to reach perfection. Not every Partial verdict needs to become a Pass. Stop iterating when:

Stop when
  • Every remaining Partial has a known root cause and fast, predictable human correction
  • Total time (agent + human review) is less than 50% of the manual process
  • The last 2 iteration cycles produced only marginal improvements
  • No critical failures in the last 3 evaluation rounds
Keep going when
  • Any test case produces a Fail—unusable or misleading output
  • The agent takes longer than manual once you include review time
  • You have untested edge cases the agent will encounter in production
  • More than 4–5 cycles without progress (signals a scope issue)

HIL data as evaluation signal

Your human-in-the-loop checkpoints produce evaluation data as a side effect of normal operation. Approvals (no changes) mean the agent got it right—track the approval rate over time. A consistently high approval rate is evidence that a checkpoint can be removed. Modifications (human edits the output) are prompt improvement candidates—each tells you where the agent’s reasoning diverged from yours. Rejections (sent back for rework) are error signals—each should go into the iteration backlog. This production data provides a richer evaluation signal than any test suite.

The graduation criteria: all 5+ test cases pass without major corrections, total time under 50% of manual, no critical failures in the last 3 rounds, at least one colleague has validated output quality, and error handling has been tested for all identified failure modes. If you can check all five, your agent graduates from prototype to production tool.

Quick self-check

Before you move on to Lesson 8, where you will build a release notes agent end-to-end, make sure these concepts are solid. These questions map directly to the testing and iteration work you will do.

1

You are testing a release notes agent and the output is missing all security patches. Walk the four layers: which layer do you check first, what do you look for, and where would you expect the failure to be?

Start at the tool layer. Is the data even reaching the prompt? Check the label filter in fetch_merged_prs before touching the prompt.

2

Your routing function sends “Amber” health scores to human_review. How would you unit test this without invoking the full graph or calling the LLM?

Construct a synthetic state dict with health_score: "Amber" and assert on the return value. No mocking needed—routing functions are pure logic.

3

After three iteration cycles, your agent produces correct output but the total time (agent execution + your review) is 60% of the manual process. Does this meet the graduation criteria? What would you do?

The 50% threshold is not met. But check which part is slow—if it is your review time, the preceding node may need improvement. If it is agent execution, consider whether a cheaper model or fewer API calls would help.

4

Name the three types of signal your HIL checkpoints produce. Which one tells you a checkpoint might be removable?

Approvals, modifications, and rejections. A consistently high approval rate at a checkpoint means the human review is not adding value—consider removing it.

Key takeaways

1. Test agents like code: unit tests for nodes (fast, free, every commit), integration tests for the graph (slower, CI only), evaluation runs against real scenarios. 2. When a test fails, walk the four layers—tool, prompt, graph, scope—in order. The first layer that failed is where you fix. 3. The iteration cycle is: run tests, diagnose with four-layer elimination, fix at the right layer, re-test for regressions, log. 4. Graduation is not perfection—it is consistent, predictable quality with a measurable time saving over the manual process.
